# 모바일 병원 탐색 UX/UI 개선 계획서

## 1. 목적

모바일에서 병원 목록을 올려 보고, 병원을 선택하고, 상세 정보를 확인하는 핵심 흐름의 불편감을 줄인다. 특히 사용자가 목록을 스크롤하려는 동작이 바텀시트 이동으로 잘못 해석되거나, 시트와 내부 콘텐츠가 서로 스크롤을 빼앗는 문제를 우선 해결한다.

이번 계획은 시각 디자인 교체보다 다음 사용자 경험 회복을 목표로 한다.

```text
지도 탐색
→ 병원 목록 올리기
→ 목록을 끊김 없이 스크롤하기
→ 병원 선택
→ 상세 정보 읽기
→ 이전 목록 위치로 돌아가기
```

## 2. 점검 범위와 한계

### 점검 범위

- 시민 화면 `MobileBottomSheet`
- 정책 화면 `AdminMobileBottomSheet`
- 병원 목록 `HospitalSidebarList`
- 병원 상세 `HospitalDetailPanel`
- 위치 안내와 진료 대상 필터
- 지도 제스처와 바텀시트 제스처의 관계
- 작은 화면, 브라우저 주소창 변화, 하단 안전영역

### 점검 근거

- 사용자가 실제 모바일 사용 중 병원 목록을 위로 올리고 스크롤하는 과정에서 높은 불편감을 보고했다.
- 현재 구현의 상태, 스크롤 컨테이너, 드래그 범위와 고정 높이를 코드로 확인했다.
- 자동 브라우저 조작 환경은 이번 점검 시점에 연결되지 않아 실제 기기별 제스처 재현은 수행하지 못했다. 아래 체감 항목은 사용자 보고와 구현 구조에서 도출한 가설이며, 구현 전 실제 기기 관찰로 확정한다.

## 3. 사람 입장에서 예상되는 불편감

### 3.1 목록을 올리려는데 시트가 움직인다

사용자는 병원 목록을 위로 스크롤하려고 손가락을 움직였지만, 현재는 바텀시트 전체에 `drag="y"`가 적용되어 있다. 내부 목록도 같은 세로축으로 스크롤되기 때문에 다음과 같은 체감을 만들 수 있다.

- 목록을 밀었는데 시트가 확장되거나 접힌다.
- 같은 동작이 시작 위치와 속도에 따라 다르게 반응한다.
- 천천히 읽으며 스크롤하기 어렵고, 더 강하게 밀어야 한다고 느낀다.
- 목록 끝에 도달하면 갑자기 시트가 끌려 내려간다.

이는 사용자의 의도와 시스템 반응이 일치하지 않는 가장 높은 우선순위 문제다.

### 3.2 드래그 가능한 영역이 명확하지 않다

화면에는 드래그 핸들이 있지만 실제로는 시트 전체가 드래그 대상이다. 사용자는 핸들만 잡아야 하는지, 헤더를 잡아도 되는지, 목록 위에서도 시트를 옮길 수 있는지 학습하기 어렵다.

`aria-label`은 `div`에만 있고 키보드로 시트 상태를 바꿀 수 있는 버튼 역할은 제공하지 않는다. 시각적 의미와 실제 상호작용 범위가 일치하지 않는다.

### 3.3 절반 높이에서 실제 목록이 너무 적게 보인다

시트는 `90dvh` 높이이며 절반 상태에서 `40dvh`만큼 아래로 이동한다. 남는 약 `50dvh` 안에 다음 요소가 모두 들어간다.

- 드래그 핸들
- 제목과 설명
- 위치 상태 안내
- 진료 대상 제목
- 가로 필터 칩
- 폴백 경고
- 병원 목록

작은 화면에서는 목록이 한두 개만 보이거나, 병원을 비교하려면 반복 스크롤이 필요할 수 있다. 사용자는 “병원 목록 화면”보다 “고정 안내 영역 아래의 작은 창”을 조작하는 느낌을 받는다.

### 3.4 스냅 위치가 화면과 콘텐츠에 적응하지 않는다

현재 위치는 `0`, `40dvh`, `calc(90dvh - 150px)`로 고정되어 있다. 다음 조건을 반영하지 않는다.

- 실제 화면 높이
- 모바일 브라우저 주소창의 확장·축소
- 상단 내비게이션 높이
- 안전영역
- 위치 경고나 데이터 폴백 배너 유무
- 사용자가 보고 있는 목록 또는 상세 콘텐츠

특히 접힌 상태의 `150px`은 실제 헤더 높이가 아니라 주석상 추정값이므로 콘텐츠가 과하게 보이거나 잘릴 수 있다.

### 3.5 상세 화면에서 이중 스크롤이 생길 수 있다

`MobileBottomSheet`의 상세 래퍼에 `overflow-y-auto`가 있고, 내부 `HospitalDetailPanel`에도 별도의 `overflow-y-auto`가 있다. 부모와 자식이 모두 스크롤 컨테이너가 되면 다음 문제가 생길 수 있다.

- 어느 영역이 움직이는지 예측하기 어렵다.
- 내부 스크롤 경계에서 움직임이 멈추거나 튄다.
- 상세를 읽다가 바텀시트 드래그가 시작된다.
- 하단 전화·길찾기 버튼까지 도달하기 어렵다.

### 3.6 목록으로 돌아오면 탐색 맥락을 잃을 수 있다

병원 선택 시 목록 대신 상세 화면으로 교체된다. 목록으로 돌아가기 버튼은 있지만 이전 목록의 스크롤 위치를 명시적으로 저장·복원하지 않는다. 브라우저의 DOM 유지 방식에 따라 우연히 유지될 수 있으나 제품 동작으로 보장되지 않는다.

사용자는 “아까 비교하던 병원이 어디였지?”라는 상황을 겪을 수 있다.

### 3.7 카드 안의 전화·길찾기와 카드 선택이 충돌할 수 있다

병원 행 전체가 `role="button"`인 `div`이고, 내부에 전화 링크와 카카오 길찾기 링크가 있다. 링크 이벤트가 부모 선택까지 전달되면 외부 이동 직전에 병원 상세가 열리거나 상태가 바뀔 수 있다. 중첩 인터랙션은 터치 사용자의 실수를 늘리고 접근성 의미도 불명확하게 만든다.

### 3.8 시민 화면과 정책 화면의 동작이 비슷하지만 별도 구현이다

두 모바일 시트가 같은 세 단계 상태, 같은 드래그 임계값, 같은 고정 위치를 각각 구현한다. 한쪽만 수정되거나 시간이 지나며 동작이 달라질 가능성이 높다.

## 4. 원인 정리

| 우선순위 | 원인 | 현재 근거 | 사용자 영향 |
|---|---|---|---|
| P0 | 시트 전체 세로 드래그와 목록 세로 스크롤 경쟁 | `motion.aside` 전체에 `drag="y"` | 목록 스크롤 실패·오작동 |
| P0 | 상세의 중첩 스크롤 컨테이너 | 외부 래퍼와 상세 패널 모두 `overflow-y-auto` | 멈춤·튐·하단 도달 어려움 |
| P1 | 고정 스냅 값과 추정 높이 | `40dvh`, `150px` 하드코딩 | 기기별 노출량 불안정 |
| P1 | 고정 영역이 목록 공간을 과도하게 차지 | 위치 안내·제목·필터가 모두 `shrink-0` | 비교 가능한 병원 수 감소 |
| P1 | 목록 위치 복원 계약 없음 | 목록과 상세 조건부 교체 | 비교 맥락 상실 |
| P1 | 카드 전체 선택과 내부 링크 중첩 | `role="button"` 내부 링크 | 오터치·접근성 혼란 |
| P2 | 시민·정책 시트 로직 중복 | 유사 코드 별도 유지 | 수정 누락·행동 불일치 |

## 5. 개선 원칙

1. 한 번의 세로 제스처는 한 컴포넌트만 소유한다.
2. 병원 목록 위에서는 목록 스크롤이 기본 동작이다.
3. 시트 높이 변경은 드래그 핸들 또는 명시적인 헤더 영역에서만 시작한다.
4. 내부 콘텐츠의 스크롤 컨테이너는 한 단계로 제한한다.
5. 스냅 위치는 화면과 안전영역을 기준으로 계산하고 추정 픽셀값을 제거한다.
6. 병원 선택 전후의 목록 위치와 필터 상태를 보존한다.
7. 시민·정책 화면은 공통 시트 컨트롤러를 사용하되 콘텐츠 표현은 분리한다.

## 6. 단계별 실행 계획

### 1단계: 실제 기기 관찰과 기준선 확정

구현 전에 Android Chrome과 iOS Safari 크기에서 다음 행동을 화면 녹화한다.

1. 접힌 시트를 절반으로 올린다.
2. 병원 목록을 1번에서 10번까지 천천히 내린다.
3. 빠르게 플릭하고 멈춘다.
4. 병원을 선택해 상세 하단까지 읽는다.
5. 목록으로 돌아가 이전 위치를 확인한다.
6. 지도 이동 후 마커를 선택하고 시트 변화를 확인한다.

기록할 항목:

- 의도하지 않은 시트 이동 횟수
- 목록 10번째 병원까지 도달하는 데 필요한 제스처 수
- 상세 하단 행동 버튼 도달 가능 여부
- 이전 목록 위치 복원 여부
- 사용자가 다음 동작을 망설인 지점

### 2단계: P0 제스처 충돌 제거

`MobileBottomSheet`와 `AdminMobileBottomSheet`에서 시트 전체 드래그를 제거하고 드래그 핸들 영역만 시트 이동을 시작하도록 변경한다.

- Framer Motion의 `useDragControls` 사용 검토
- 시트 본체는 `dragListener={false}`
- 핸들의 포인터 시작 시에만 `dragControls.start(event)`
- 목록과 상세 콘텐츠 영역에는 기본 브라우저 세로 스크롤 허용
- 목록 스크롤 위치가 최상단일 때도 아래 스와이프가 자동으로 시트 접기로 전환되지 않게 함
- 핸들을 실제 `button`으로 만들고 탭하면 다음 스냅 위치로 이동
- `aria-expanded`, 현재 상태 설명, 최소 44px 터치 영역 적용

완료 기준:

- 목록 어느 지점에서 스크롤해도 시트 위치가 바뀌지 않는다.
- 핸들에서는 세 단계 이동이 정상 동작한다.
- 지도, 시트 핸들, 콘텐츠의 제스처 소유권이 분명하다.

### 3단계: 스크롤 컨테이너 단일화

목록 화면과 상세 화면마다 실제 세로 스크롤 소유자를 하나만 둔다.

- 목록: `HospitalSidebarList`의 `ul` 또는 상위 콘텐츠 영역 중 하나만 스크롤
- 상세: 모바일에서는 바깥 래퍼 또는 `HospitalDetailPanel` 중 하나만 스크롤
- 상세 패널에 모바일·데스크톱 `variant`를 추가해 높이와 overflow 책임 분리 검토
- `min-h-0`, `flex-1`, `overflow-hidden` 경계를 명시적으로 정리
- 하단 여백은 실제 모바일 내비게이션과 `safe-area-inset-bottom`을 합산

완료 기준:

- 상세를 위에서 아래까지 한 번의 연속된 스크롤로 읽을 수 있다.
- 스크롤 경계에서 다른 컨테이너로 갑자기 전환되지 않는다.

### 4단계: 스냅 위치와 정보 밀도 개선

고정 `150px` 추정값을 없애고 다음 세 상태를 사용자 목적에 맞게 정의한다.

- `peek`: 지도 중심, 병원 수와 첫 추천만 짧게 노출
- `list`: 목록 탐색 중심, 화면의 약 55~65% 사용
- `full`: 상세 읽기 또는 긴 목록 탐색, 상단 안전영역 아래까지 확장

구현 시 CSS 변수 또는 측정된 헤더 높이를 사용하고 `dvh`, 상단 내비게이션, 안전영역을 함께 반영한다. 배너 유무로 헤더가 길어져도 목록 영역이 0에 가까워지지 않도록 최소 높이를 둔다.

고정 영역도 축소한다.

- 위치 안내는 한 줄 상태로 압축하고 오류일 때만 확장
- `누가 진료받나요?` 제목과 칩을 한 행으로 재배치 검토
- 폴백 경고는 요약 한 줄과 펼치기 방식 검토
- 목록 상단에 현재 정렬 기준과 결과 수만 유지

### 5단계: 목록 탐색 맥락 보존

- 목록 스크롤 컨테이너 ref와 `scrollTop`을 상태로 보존
- 상세에서 목록으로 돌아올 때 다음 프레임에 위치 복원
- 선택했던 병원 행에 짧은 강조 또는 `aria-current` 적용
- 필터 변경으로 선택 병원이 사라질 때 이유를 짧게 안내
- 지도 마커 선택 시 해당 병원 행을 중앙이 아니라 최소 이동으로 노출

완료 기준:

- 10번째 병원을 선택한 뒤 돌아오면 10번째 병원 주변이 그대로 보인다.

### 6단계: 병원 카드의 행동 분리

행 전체를 가짜 버튼으로 만들지 않고 의미를 분리한다.

- 병원명·상태 영역: 상세 열기 버튼
- 전화: 독립 링크
- 길찾기: 독립 링크
- 내부 링크 클릭이 상세 선택으로 전파되지 않도록 이벤트 계약 확인
- 주요 행동의 터치 영역 최소 44×44px 확보
- 거리·ETA·병상·병원명 순으로 정보 위계 단순화

### 7단계: 공통 바텀시트 컨트롤러 추출

P0·P1 동작이 안정된 뒤 시민·정책 시트의 상태와 제스처 로직을 공통 훅 또는 Container로 분리한다.

예상 책임:

- 시트 상태와 전환
- 스냅 위치 계산
- 드래그 종료 판정
- 핸들 키보드·탭 동작
- 선택 또는 상세 진입 시 자동 확장 규칙

시민·정책 콘텐츠와 데이터 변환은 각각의 Container에 유지하고, 공통 컨트롤러는 표현 데이터를 알지 않게 한다.

## 7. 변경 예상 파일

| 파일 | 변경 목적 |
|---|---|
| `frontend/src/widgets/map-dashboard/MobileBottomSheet.tsx` | 시민 시트 제스처·스냅·스크롤 책임 정리 |
| `frontend/src/widgets/app/AdminMobileBottomSheet.tsx` | 정책 시트에 같은 행동 규칙 적용 |
| `frontend/src/widgets/map-dashboard/HospitalSidebarList.tsx` | 목록 스크롤 위치 보존과 카드 행동 분리 |
| `frontend/src/widgets/map-dashboard/HospitalSidebarControls.tsx` | 고정 영역 높이와 필터 조작성 개선 |
| `frontend/src/widgets/map-dashboard/HospitalDetailPanel.tsx` | 모바일 상세 스크롤 단일화 |
| `frontend/src/widgets/map-dashboard/lib/*` | 공통 시트 상태·스냅 컨트롤러 후보 |
| `frontend/tests/e2e/*` | 모바일 제스처·목록 복원 회귀 테스트 |

새 컴포넌트나 훅을 만들기 전에 `shared/`와 기존 바텀시트 구현을 다시 검색한다. 공통화는 제스처 동작이 검증된 뒤 수행해 잘못된 추상화를 피한다.

## 8. 검증 계획

### 단위 테스트

- 현재 상태와 드래그 방향에 따른 다음 스냅 상태
- 임계 속도·거리 경계값
- 상세 진입 시 필요한 자동 확장 규칙
- 목록 스크롤 위치 저장·복원

### 컴포넌트·통합 테스트

- 콘텐츠 영역 포인터 동작이 시트 드래그를 시작하지 않음
- 핸들 동작만 시트 상태를 변경함
- 필터 변경 후 결과와 선택 상태가 일치함
- 전화·길찾기 클릭이 병원 선택을 중복 실행하지 않음

### 모바일 E2E

최소 두 뷰포트에서 확인한다.

- 390×844: 일반 스마트폰 기준
- 360×640: 작은 화면 스트레스 기준

핵심 시나리오:

1. 핸들로 `peek → list → full` 이동
2. 목록을 스크롤하는 동안 시트 위치 유지
3. 10번째 병원 선택 후 목록 위치 복원
4. 상세 하단 전화·길찾기 도달
5. 지도 드래그가 시트나 목록을 움직이지 않음
6. 화면 회전 또는 뷰포트 높이 변화 후 시트가 화면 밖으로 벗어나지 않음

### 실제 기기 확인

- Android Chrome
- iOS Safari 또는 iOS WebView
- 한 손 엄지 조작
- 브라우저 주소창이 접히고 펼쳐지는 상황
- 느린 스크롤, 빠른 플릭, 스크롤 경계 동작

## 9. 성공 지표

| 지표 | 목표 |
|---|---:|
| 목록 스크롤 중 의도하지 않은 시트 상태 변경 | 0회 |
| 10번째 병원까지 도달하는 제스처 | 3회 이내 |
| 상세 하단 핵심 행동 도달 성공률 | 100% |
| 상세에서 목록 복귀 후 위치 유지 | 100% |
| 핸들·전화·길찾기 터치 영역 | 최소 44×44px |
| 모바일 E2E 핵심 시나리오 | 전부 통과 |

정량 지표와 함께 실제 사용자에게 “목록을 올리고 병원 하나를 골라 상세 하단까지 확인해 달라”고 요청한 뒤, 조작 방법을 묻지 않고 완료할 수 있는지도 관찰한다.

## 10. 작업 순서와 우선순위

1. **P0:** 드래그 핸들 전용화, 목록·상세 스크롤 충돌 제거
2. **P0:** 상세 중첩 스크롤 단일화
3. **P1:** 스냅 위치와 고정 영역 높이 개선
4. **P1:** 목록 위치 복원과 병원 카드 행동 분리
5. **P1:** 작은 화면·안전영역·주소창 변화 검증
6. **P2:** 시민·정책 공통 시트 컨트롤러 추출
7. **P2:** 시각적 세부 조정과 애니메이션 튜닝

시각적 리디자인보다 제스처 충돌과 스크롤 소유권을 먼저 해결한다. 사용자가 목록을 안정적으로 움직일 수 없는 상태에서는 색상, 그림자, 카드 모양을 바꿔도 체감 UX가 개선되지 않는다.

## 11. 최종 완료 조건

- 병원 목록을 스크롤하는 동작과 시트 높이를 바꾸는 동작이 명확히 분리된다.
- 목록과 상세 화면에 각각 하나의 세로 스크롤 컨테이너만 존재한다.
- 작은 화면에서도 병원 목록을 비교할 수 있는 최소 공간이 보장된다.
- 상세를 보고 돌아왔을 때 이전 목록 위치와 필터가 유지된다.
- 시민·정책 화면이 동일한 시트 행동 규칙을 따른다.
- 타입 검사, 단위 테스트, 빌드, 모바일 E2E와 실제 기기 점검 결과가 기록된다.
