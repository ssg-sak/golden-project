# 🚀 외부 API 이원화(Dualization) 구현 설명서

> **대상 독자:** 코딩을 막 시작한 부트캠프 1달차 비전공자 수강생  
> **주제:** "왜 API를 이원화(두 갈래로 나누기) 했으며, 코드에서 어떻게 구현했는가?"

---

## 1. API 이원화가 무엇인가요?

**API 이원화**란 똑같은 목적(예: 실시간 응급실 병상 수 가져오기)을 달성하기 위해, **두 개 이상의 다른 데이터 제공처**를 준비해두고 상황에 맞게 스위치처럼 바꿔가며 쓰는 기술을 말합니다.

*   **배달 앱 비유:** 치킨을 시킬 때 '배달의민족' 서버가 터지면 '요기요'를 켜서 시키는 것과 같습니다.
*   **우리 프로젝트 적용:** 
    *   플랜 A: **공공데이터포털** (`DATA_GO_KR`)
    *   플랜 B: **재난안전데이터공유플랫폼** (`SAFETY_DATA`)

`.env` 파일(환경 설정 파일)에서 `BED_API_PROVIDER=SAFETY_DATA`라고 단어 하나만 바꿔주면, 코드 수정 없이 백엔드가 알아서 데이터를 가져오는 목적지를 바꿉니다.

---

## 2. 이걸 왜 하는 건가요? (포트폴리오 핵심 포인트 ⭐)

면접관이 **"왜 굳이 API를 이원화했나요?"** 라고 묻는다면 이렇게 대답할 수 있습니다.

> *"공공 API는 가끔 점검을 하거나 트래픽이 몰려 서버가 다운되는 경우가 있습니다. 응급 의료 서비스는 골든타임이 생명이므로, 하나의 데이터 제공처가 막히더라도 다른 제공처에서 데이터를 끌어올 수 있도록 **유연하고 안정적인(장애 내성이 있는) 백엔드 구조**를 설계하고 싶었습니다."*

---

## 3. 우리 코드에서는 어떻게 작동하나요?

`backend/app/services/hospital_realtime.py` 파일을 보면 크게 **3가지 핵심 문제**를 해결한 것을 볼 수 있습니다.

### ① 어디로 요청을 보낼지 결정하기 (if - else)
파이썬의 가장 기초적인 `if - else` 문법을 사용합니다.
```python
provider = get_bed_api_provider() # .env 파일에서 현재 어떤 설정인지 읽어옴

if provider == "SAFETY_DATA":
    # 1. 재난데이터포털 주소(URL)로 요청을 보냄
    url = "https://www.safetydata.go.kr/V2/api/DSSP-IF-00242"
else:
    # 2. 기존 공공데이터포털 주소로 요청을 보냄
    url = f"{BED_API_BASE}/{BED_OPERATION}"
```

### ② 데이터 형식(XML vs JSON)의 차이 극복하기
데이터 제공처가 다르면, 그들이 데이터를 포장해서 보내주는 형식(택배 박스 모양)도 다릅니다.
*   **공공데이터포털:** `XML` 형식 (과거에 많이 쓰던 복잡한 태그 방식 `<hvec>10</hvec>`)
*   **재난데이터포털:** `JSON` 형식 (요즘 많이 쓰는 딕셔너리 형태 `{"EMRO": 10}`)

그래서 코드를 보면 `response.json()` 이라는 함수를 써서 파이썬이 쉽게 읽을 수 있는 딕셔너리 형태로 껍질을 벗겨내는 작업이 추가되었습니다.

### ③ 이름표 짝맞추기 (데이터 매핑)
여기가 가장 중요하고 재미있는 부분입니다! 서로 다른 두 회사가 데이터를 주다 보니 **단어(Key)** 를 다르게 씁니다.

1.  **병상 수 이름이 다름:**
    *   A회사(공공)는 응급실 병상 수를 `hvec`이라고 부릅니다.
    *   B회사(재난)는 응급실 병상 수를 `EMRO`라고 부릅니다.
    *   **해결:** 코드에서 `row.get("EMRO", "0")` 처럼 B회사의 단어를 찾아 우리 시스템이 이해하는 변수(`hvec`)에 담아줍니다.
2.  **병원 이름이 안 옴:**
    *   B회사(재난)는 응급실 데이터를 줄 때 "대구파티마병원"이라는 이름을 주지 않고, **`A2800010`** 이라는 암호 같은 병원 고유 ID(`BFR_INST_ID`)만 던져줍니다.
    *   **해결:** `hospital_mapping.py` 파일에 `{"A2800010": "대구파티마병원"}` 처럼 **사전(Dictionary)** 을 만들어 두었습니다. 코드가 ID를 받으면 이 사전을 뒤져서 진짜 이름으로 번역해 줍니다!

## 4. 혹시라도 두 군데 다 서버가 터진다면? (예외 처리)

코드를 보면 `try ... except Exception as exc:` 라는 블록이 많이 보일 것입니다.
이것은 **"에러가 나면 프로그램 전체를 죽이지 말고(크래시 내지 말고), 조용히 빠져나와라"** 라는 뜻입니다.

데이터를 가져오다 실패하면, 백엔드는 에러 화면을 띄우는 대신 `null_bed_payload()` 라는 함수를 호출해서 **병상 데이터: '알 수 없음(null)'** 이라는 빈 껍데기를 프론트엔드로 보냅니다. 
그러면 사용자 화면에는 병원 목록과 지도는 그대로 예쁘게 보이면서, 병상 수 부분만 회색으로 **"실시간 확인 중"** 이라고 뜨게 됩니다. 

부트캠프 수강생으로서 이 부분을 **"Graceful Degradation(우아한 성능 저하) 원칙을 적용하여 예외 처리를 했습니다"** 라고 설명하시면 아주 훌륭한 포인트가 됩니다!

---

## 5. 전체 데이터 흐름과 컴포넌트 구조 (백엔드 ➔ 프론트엔드)

이원화된 API 데이터가 어떻게 사용자 화면까지 전달되는지 **컴포넌트 구조와 데이터 흐름**을 정리했습니다.

### 🌊 데이터 흐름도 (Data Flow)

**외부 API (공공/재난 포털)** ➔ **백엔드 (FastAPI)** ➔ **프론트엔드 전역 상태 (Zustand)** ➔ **UI 컴포넌트 (React)**

### 🧱 핵심 파일 및 컴포넌트 역할

#### [백엔드 (Python/FastAPI)]
*   **`backend/app/services/hospital_realtime.py` (컨트롤 타워)**
    *   `.env` 설정을 읽어 공공/재난 API 중 어디로 갈지 분기(`if-else`) 처리합니다.
    *   외부 API에서 받은 각기 다른 응답(XML 또는 JSON)을 파싱하여, 우리 서비스만의 공통 규격(`available_beds`)으로 포장합니다.
*   **`backend/app/services/hospital_mapping.py` (번역기)**
    *   재난데이터포털이 보내주는 병원 암호(예: `A2800010`)를 "대구파티마병원"이라는 진짜 이름으로 번역(`HPID_TO_NAME`)해줍니다.

#### [프론트엔드 (React/TypeScript)]
*   **`frontend/src/shared/store/hospitalStore.ts` (데이터 보관소 - Zustand)**
    *   백엔드에서 가공해준 최종 병상 데이터를 API로 불러와, 프론트엔드 어디서든 꺼내 쓸 수 있도록 **전역 상태(Global State)** 로 저장합니다.
*   **`frontend/src/widgets/landing/BedStatusBadge.tsx` (UI 시각화 컴포넌트)**
    *   가용 병상 수에 따라 초록색(진료 가능) 또는 빨간색(수용 불가) 뱃지를 띄웁니다.
    *   만약 두 외부 API가 모두 터져서 백엔드가 `null`(알 수 없음) 값을 보내더라도 이 컴포넌트가 다운되지 않고, **"실시간 확인 중"** 이라는 회색 뱃지를 보여주도록 방어 로직이 적용되어 있습니다.

> *"아니요, **백엔드에서 외부 API의 차이(XML/JSON, 필드명 등)를 모두 흡수**하여 동일한 규격으로 데이터를 가공해 주었기 때문에, 프론트엔드 컴포넌트(`BedStatusBadge`)는 데이터 출처가 바뀌었는지 모른 채 **기존 UI 로직을 그대로 재사용**할 수 있었습니다. 이를 통해 프론트엔드와 백엔드의 결합도를 낮추는 좋은 구조를 경험했습니다."*
