# 대구 골든타임 데이터 무결성·갱신 안전성 강화 보고서

- 작성일: 2026-07-18
- 대상 분석 릴리스: `2026-07-18-r2`
- 대상 범위: 시민용 병상 캐시, 정책 데이터 갱신, 25개 기관 정책 파이프라인, 정책 화면 상태 안내
- 보고서 성격: 구현·검증 기록

## 1. 작업 배경

24개 기관 기준의 과거 분석에서 기관 한 곳이 누락된 사실이 확인된 뒤, 현재 정책 분석은 25개 기관 기준으로 전면 재실행됐다. 이후 외부 평가에서 다음 위험이 추가로 지적됐다.

1. 전체 인구를 연령별 취약인구로 추정할 수 있는 자동 갱신 경로
2. 관리자 갱신 토큰 미설정 시 요청을 허용하는 fail-open 인증
3. 현재 후보 정본 9곳과 과거 후보 추적 5곳의 불일치
4. 백엔드의 재분석 대기 상태를 프론트가 `false`로 덮어쓰는 문제
5. 갱신 실패·지연과 마지막 정상 분석 시각에 대한 사용자 안내 부족
6. 위험 점수를 고정 절대구간처럼 보이게 하는 화면 표현
7. 스케줄러 실행 증빙, 병상 캐시 신선도, 도로 경로 계보 부족

이번 작업은 현재 검증본을 임의로 덮어쓰지 않으면서 위 결함을 차단하고, 분석 입력부터 공개 화면까지 동일한 데이터 계약을 적용하는 것을 목표로 했다.

## 2. 최종 판정

즉시 수정 가능한 데이터 무결성·보안·표시 결함은 모두 수정했다. 현재 `2026-07-18-r2`는 다음 계약을 만족한다.

|---|---:|
| 행정동 | 150곳 |
| 응급 관련 기관 | 25곳 |
| 소아 기준 기관 | 6곳 |
| 고령층 기준 기관 | 19곳 |
| 정책 후보 정본 | 9곳 |
| 정책 후보 추적 | 9곳 |

후보 정본과 후보 추적은 `mode`, `id`, 위도, 경도를 소수점 일곱 자리까지 비교해 일치함을 확인했다. 도로 행렬과 입지 최적화 결과도 입력 해시와 경로 결과 해시가 모두 일치한다.

## 3. 주요 결함과 해결 내용

### 3.1 연령별 인구 추정 제거

`PopulationSnapshot`은 전체 인구만 저장하지만 과거 내보내기 코드는 기존 0~9세·65세 이상 비율을 전체 인구에 곱해 새 분석 입력을 만들 수 있었다. 이 경로를 제거했다.

현재 동작은 다음과 같다.

1. 공식 연령별 CSV에 `동이름`, `65세이상_인구`, `0~9세_인구`가 모두 있어야 한다.
2. 행정동 150개, 행정동명 중복 없음, 음수·결측 없음 조건을 검사한다.
3. 공식 연령별 입력의 기준월과 현재 분석 기준월이 다르면 재분석을 중단한다.
4. 전체 인구 DB 값을 연령별 인구로 추정해 CSV를 덮어쓰지 않는다.
5. 실패하면 기존 검증 분석본을 유지한다.

관련 구현: `backend/app/services/pipeline.py`

### 3.2 관리자 갱신 API fail-closed

관리자 토큰이 서버에 설정되지 않은 경우 갱신 API를 `503`으로 비활성화한다. 서버 토큰은 있지만 요청 토큰이 없거나 다르면 상수시간 비교 후 `401`을 반환한다.

관련 구현: `backend/app/api/routes/dashboard.py`

### 3.3 후보 정본과 추적 자료 9대9 일치

파이프라인 실행 순서를 다음처럼 변경했다.

```text
공간 전처리
→ 후보 민감도 분석
→ 안정 후보 9곳 생성
→ 안정 후보 9곳 추적
→ 도로 경로·입지 최적화
→ 시각화
→ 단일 정책 릴리스 검증·생성
```

정책 릴리스 생성 전 후보 정본과 추적 자료의 개수, 모드, ID, 좌표가 일치하지 않으면 실행을 실패시킨다. 후보 추적 파일의 SHA-256도 릴리스에 기록한다.

관련 구현:

- `ai-model/build_stable_policy_candidates.py`
- `ai-model/build_accessibility_candidate_trace.py`
- `ai-model/run_integrated_policy_pipeline.py`
- `ai-model/build_policy_release.py`

### 3.4 프론트 갱신 상태 투명성

정책 릴리스가 존재한다는 이유만으로 `pending=false`를 강제하던 로직을 제거했다. 접힌 상세 영역을 열지 않아도 다음 상태가 항상 보인다.

- 원천자료 변경으로 재분석 대기 중
- 공공데이터 갱신 지연
- 상태 API 조회 실패
- 현재 화면이 이전 검증 분석본임
- 마지막 정상 분석본 확인 시각

관련 구현: `frontend/src/widgets/map-dashboard/PolicyDataPipeline.tsx`

### 3.5 상대 위험 기준으로 화면 통일

`1,500`, `5,000`, `10,000`을 고정 의료 위험구간처럼 표시하던 UI를 제거했다. 현재 경계값은 150개 행정동 VDI를 높은 순서로 정렬했을 때 상위 25%를 우선 확인 대상으로 구분하는 상대값이다.

추가로 지도 슬라이더의 최대값을 `10,000`으로 고정하던 문제를 수정했다. 슬라이더는 현재 분석 데이터의 최솟값과 최댓값을 사용한다. 선택 동네 상세 화면도 현재 상대 경계값을 기준으로 `상위 25% 우선 확인` 여부만 표시한다.

관련 구현:

- `frontend/src/widgets/map-dashboard/DashboardStatsBar.tsx`
- `frontend/src/widgets/map-dashboard/MetricsGuide.tsx`
- `frontend/src/widgets/map-dashboard/MapToolbar.tsx`
- `frontend/src/widgets/map-dashboard/VulnerabilityDistrictView.tsx`

### 3.6 병상 캐시 신선도 전달

병원 API를 조회한 시각을 병상 원천 갱신 시각처럼 표시하지 않도록 수정했다. 백엔드는 병상 캐시 생성 시각과 stale 여부를 응답 헤더로 전달하고, 프론트는 캐시가 오래됐거나 마지막 갱신이 실패한 경우 이전 자료임을 표시한다.

관련 구현:

- `backend/app/services/bed_cache.py`
- `backend/app/api/routes/hospitals.py`
- `frontend/src/data/api/hospitals.ts`
- `frontend/src/shared/store/hospitalStore.ts`

### 3.7 스케줄러 운영 증빙

예약 작업은 시작, 성공, 실패, 소요시간, 변경 여부, 재분석 대기 여부를 로그로 남긴다. APScheduler 작업별 `max_instances=1`, coalescing, misfire 유예를 설정했고 애플리케이션 DB 작업 잠금도 유지한다.

인구 확인은 매일 실행하지 않고 월 단위로 조정했다. 스케줄러는 원천 수집·변경 감지만 담당하며 새 정책 릴리스를 자동 공개하지 않는다.

관련 구현: `backend/app/services/scheduler.py`

### 3.8 도로 경로 계보 강화

도로 행렬 메타데이터에 다음 정보를 추가했다.

- 행렬 생성 시각
- 캐시 전용 또는 API·캐시 혼합 실행 모드
- 기존 캐시 사용 경로 수
- 신규 조회 경로 수
- 좌표 보정 경로 수
- 입력 데이터 SHA-256
- 실제 도로 결과 SHA-256

입지 최적화와 정책 릴리스는 동일한 도로 결과 해시를 참조해야 한다.

현재 검증 실행은 기존 캐시 5,100건을 사용한 오프라인 재현 실행이며, 좌표 보정 기록이 있는 경로는 460건이다. 좌표 보정은 API가 원 좌표를 경로에 연결하지 못한 경우의 재시도 기록이며, 실제 구급 이송 경로를 의미하지 않는다.

관련 구현: `ai-model/build_actual_road_accessibility.py`

### 3.9 레거시 최적입지 공개 경로 제거

프론트 정책 화면은 단일 정책 릴리스의 후보 9곳만 사용한다. 백엔드 `/api/optimal-locations`도 과거 `optimal_locations.json` 대신 현재 정책 릴리스의 후보 9곳을 반환하도록 변경했다.

DB 초기 상태 역시 과거 소아·고령 후보 파일 대신 현재 안정 후보 파일과 단일 정책 릴리스를 데이터 출처로 기록한다. 기존 레거시 상태 레코드는 초기화 과정에서 제거한다.

관련 구현:

- `backend/app/api/routes/optimal_locations.py`
- `backend/app/services/data_seed.py`

### 3.10 SGIS 데이터 안전 경계

SGIS 정규화 과정에서 상위 행정코드, 전체 주소, geometry 필드가 유실되던 문제를 수정했다. SGIS 좌표가 WGS84 대구 범위로 확인될 때만 위·경도 컬럼에 저장하고, 투영좌표는 위·경도로 오인해 저장하지 않는다.

현재 SGIS stage API는 행정동 목록·변경 감시에 사용한다. 정책 공간분석 경계는 저장된 검증 GeoJSON을 사용하며 SGIS 응답을 자동 공개 경계로 승격하지 않는다.

관련 구현: `backend/app/services/fetchers/sgis.py`

## 4. 실패 안전성

정책 파이프라인은 실행 전 기존 공개 산출물을 임시 백업한다. 1~7단계 중 하나라도 실패하면 새 산출물을 공개하지 않고 기존 파일을 복원한다.

DB에서 새 병원 정본을 내보낸 뒤 공식 연령별 인구 검증 또는 통합 분석이 실패하는 경우에도, 병원 분석 입력 파일을 실행 전 바이트로 복원한다. 공개 결과만 롤백되고 입력 파일이 새 값으로 남는 불일치를 방지한다.

다음 조건은 공개 실패로 처리한다.

- 기관 수가 25곳이 아님
- 필수 기관 누락 또는 기관명 중복
- 행정동이 150곳이 아님
- 후보가 9곳이 아님
- 후보 정본과 추적의 ID·좌표 불일치
- 요청·성공 경로가 5,100건이 아님
- 누락 경로 존재
- 도로 입력·결과 해시 불일치
- 분석 버전 불일치
- 공식 연령별 인구 입력 검증 실패

## 6. 현재 공개 해석 원칙

1. 병상 정보는 준실시간 캐시 정보이며 실제 수용 가능 여부를 확정하지 않는다.
2. 조회 실패·null은 진료 불가나 수용 불가가 아니다.
3. 정책 위험 경계값은 현재 분석본 내부의 상대 기준이다.
4. 후보지는 병원 신설 확정지가 아니라 현장조사 전 검토 대상이다.
5. 도로 ETA는 일반 차량 경로이며 119 관제·구급 이송시간이 아니다.
6. 고정 PDF는 작성 당시 기록이며 최신 웹 분석과 혼용하지 않는다.

## 8. 결론

이번 개선으로 현재 r2의 25개 기관, 후보 9곳, 도로 경로 5,100건은 하나의 검증 계약과 해시 계보로 연결됐다. 자동 갱신은 전체 인구나 SGIS 목록이 바뀌었다는 이유만으로 정책 결과를 덮어쓰지 않으며, 프론트는 갱신 지연과 재분석 대기 상태를 사용자에게 숨기지 않는다.

프로젝트의 다음 운영 단계는 기능 추가가 아니라 Render 예약 실행 로그, 새 25개 기관 PDF, 실제 브라우저 시각 QA를 확보하는 것이다.
