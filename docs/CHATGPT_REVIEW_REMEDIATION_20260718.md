## 1. 작성 목적

이 문서는 대구 골든타임 프로젝트에 대한 ChatGPT 독립 기술심사 결과인 **병합 보류 78/100**을 바탕으로, 지적사항을 실제 결함과 심사자료 누락으로 구분하고 후속 조치와 검증 결과를 기록한다.

이번 개선의 목표는 점수 자체를 높이는 것이 아니라 다음 질문에 저장소 근거로 답할 수 있게 하는 것이다.

- 정책 화면의 25개 기관·9개 후보·5,100개 경로가 하나의 검증된 분석본인가?
- 인구 기준월과 민감도 결과가 실제 입력 파일에 결합되어 있는가?
- 좌표 보정이 결과를 왜곡할 수 있는 범위가 기록되고 제한되는가?
- 분석 실패 시 어떤 파일이 복원되며, 복구가 보장되지 않는 실패는 무엇인가?
- 시민 화면이 미확인 정보를 진료·수용 가능으로 오해시키지 않는가?
- 문서에 적은 안전 기능을 실제 코드와 테스트로 제시할 수 있는가?

## 2. 심사 결과의 핵심 진단

심사에서 150개 행정동, 25개 기관, 9개 후보, 후보 추적 9개, 5,100개 도로 경로와 기존 SHA-256 연결은 일관된 것으로 확인됐다. 반면 다음 문제가 병합 보류 사유가 됐다.

1. 인구 기준월 `2026.06`이 릴리스 생성 코드에 고정돼 있었다.
2. 민감도 분석의 240회가 전체인지 모드별인지 불명확했다.
3. 최종 후보 이전의 민감도 원본이 릴리스 계보에 포함되지 않았다.
4. 좌표 보정 경로 460건은 개수만 있고 거리와 대상 목록이 없었다.
5. 롤백 설명이 실제 프로그램 내부 복원 범위보다 넓게 읽힐 수 있었다.
6. 시민 안내의 “빠른 물리적 이송 목적” 표현이 서비스 역할을 과장할 수 있었다.
7. EDA가 VDI 산식 구성요소의 상관을 독립적인 발견처럼 읽히게 할 수 있었다.
8. 생성된 Notebook 자체에는 분석 계약 검사가 없었고 Linux 한글 폰트 대안도 없었다.
9. 최초 심사 ZIP에 핵심 백엔드·프론트·테스트가 빠져 이미 구현된 안전 기능도 확인 불가로 판정됐다.

## 4. 후속 개선 내용

### 4.1 인구 기준월과 원본 파일 결합

`data/raw/population/daegu_population_real.manifest.json`을 추가했다. manifest는 다음을 기록한다.

- 인구 기준월
- 예상 행정동 수
- 원본 CSV 파일명
- 원본 CSV SHA-256
- 검증일
- 개별 원천 수집일 확인 상태

공간분석 단계는 manifest의 원본 파일명과 SHA-256을 실제 CSV와 대조한다. 행정동 수가 150개가 아니거나 해시가 다르면 분석을 중단한다. 생성된 취약성 GeoJSON에도 기준월과 인구 원본 해시를 기록한다.

정책 릴리스 생성 단계는 manifest, CSV, 취약성 GeoJSON의 기준월·행정동 수·원본 해시를 교차 검증한다. 따라서 기준월 문자열만 바꾸거나 다른 월의 GeoJSON을 섞은 릴리스는 생성되지 않는다.

### 4.2 민감도 분석 계약 강화

민감도 조건은 다음 조합이다.

```text
K 4개 × 난수 시드 5개 × 거리 상한 4개 × 군위 처리 3개 = 모드당 240개
```

소아와 어르신을 각각 실행하므로 총 실행량은 480회다. 릴리스 메타데이터에는 다음을 기록한다.

- 민감도 원본 파일 SHA-256
- 소아 전체 조건 수와 완료 수: 240/240
- 어르신 전체 조건 수와 완료 수: 240/240
- 정규화된 민감도 payload SHA-256

두 모드가 모두 240회를 완료하지 않으면 릴리스 생성을 실패시킨다.

### 4.3 좌표 보정 감사 계약

카카오 경로 API가 원 좌표를 연결하지 못한 경로의 재시도 범위를 축소하고 감사자료를 추가했다.

현재 검증본 결과:

| 항목 | 값 |
|---|---:|
| 전체 도로 경로 | 5,100건 |
| 좌표 보정 기록 경로 | 460건 |
| 경로별 보정 거리 최댓값의 평균 | 0.311321km |
| 전체 최대 보정 거리 | 0.573040km |
| 허용 상한 | 0.75km |

행렬에는 보정된 출발지·도착지 ID, 오프셋, 출발지·도착지 추정 보정 거리와 경로별 최댓값을 기록한다. 허용 상한 0.75km를 넘으면 성공 경로로 인정하지 않고 릴리스 검증을 실패시킨다.

좌표 보정은 API 연결을 위한 기술적 재시도 기록이다. 실제 구급차 이송 경로나 응급의료 접근성의 임상적 정확성을 뜻하지 않는다.

### 4.4 실패 복원 범위

통합 실행기는 시작 전에 다음 파일을 임시 백업한다.

- 공개·분석 정책 산출물
- 도로 ETA 캐시
- 최종 병원 분석 입력

실행 중 포착된 예외가 발생하면 해당 파일을 실행 전 상태로 복원한다. 백엔드 갱신 래퍼도 DB에서 병원 파일을 새로 내보내기 전 바이트를 별도로 보관하고, 인구 검증 또는 통합 분석 실패 시 복원한다.

이 복구는 프로그램이 예외를 포착하고 정리 절차를 수행할 수 있을 때의 파일 복원이다. 운영체제 강제 종료, 프로세스 강제 중단, 저장장치 장애와 여러 파일의 완전한 원자적 교체까지 보장하지 않는다. 운영 환경에서는 별도 배포 아티팩트 또는 파일시스템 스냅샷이 필요하다.

### 4.5 시민 안전 문구와 상태 시각화

시민 공통 안내를 다음 원칙으로 수정했다.

> 이 서비스는 병원 탐색을 돕는 참고 도구이며 응급 이송, 진료 가능 여부 또는 수용 가능 여부를 결정하지 않습니다. 위급할 때는 119, 의료상담은 1339, 방문 전에는 의료기관에 확인하세요.

정책 화면에서는 브라우저가 상태를 확인한 시각을 분석 생성 시각처럼 표시하지 않도록 다음 개념을 구분한다.

- 병원 상태 확인 시각
- 분석 파일 확인 시각
- 사용자 표시 분석본
- 원천자료 변경 여부
- 재분석 대기 또는 이전 검증본 유지 상태

### 4.6 EDA와 Notebook 재현성

EDA의 상관관계 단원을 `VDI 산식 구성요소와 민감도 점검`으로 바꿨다. VDI가 `ln(1 + 일반 차량 ETA) × 취약인구`로 정의되므로 구성요소와 VDI의 상관은 독립적인 인과 발견이 아니라 산식 구조와 현재 분포를 함께 반영한다는 점을 명시했다.

차트 생성기는 사용 가능한 한글 글꼴을 운영체제별로 탐색하며, 한글 글꼴이 없으면 DejaVu Sans와 영문 축·제목을 사용한다. 생성된 두 Notebook도 직접 다음 계약을 검사한다.

- 행정동 150개
- 기관 25개
- 후보 9개
- 경로 5,100개와 누락 0건
- 병원·취약성·후보·후보 추적·최적화 payload SHA-256

## 6. 심사자료 개선

첫 심사 ZIP은 분석 핵심 파일 14개만 포함해 백엔드와 시민 화면의 안전 기능이 확인 불가로 판정됐다. 후속 심사 묶음은 다음을 함께 제공한다.

- 정책 릴리스와 원천 manifest
- 민감도·도로 행렬·입지 최적화 결과
- 정책 파이프라인 전체 핵심 코드
- 관리자 인증·병상 캐시·상태 API·스케줄러·작업 잠금 코드
- 시민 병상 변환·상태 저장·정책 상태 표시 코드
- 분석·백엔드·프론트 테스트
- EDA 스크립트와 Notebook
- 검증 결과와 재평가 질문

이 조치는 기능을 새로 구현한 것과 기존 구현의 증거를 보완한 것을 구분하기 위한 것이다.

## 7. 남은 외부 확인과 한계

코드만으로 완료할 수 없는 다음 항목은 운영 계정과 실제 배포 화면에서 확인해야 한다.

1. Render에 `DATA_REFRESH_ADMIN_TOKEN`이 설정돼 있는지 확인
2. Render가 단일 인스턴스·단일 worker인지 확인
3. 스케줄러 OFF 상태를 유지한 제한적 수동 실행 로그 확보
4. 배포 시민 화면에서 병상 null·0·stale 상태 캡처
5. 구형 후보 JSON과 24개 기관 기준 PDF가 현행 자료로 노출되지 않는지 확인
6. 25개 기관 정책 PDF의 표·수치·각주를 단일 정책 릴리스와 최종 대조
7. 좌표 보정 경로가 후보 순위와 최적 조합에 미치는 추가 민감도 검토

또한 다음 분석 한계는 이번 코드 보강으로 제거되지 않는다.

- 일반 차량 ETA는 실제 119 구급차 이송시간이 아니다.
- 병상 조회값은 실제 수용 가능 여부를 확정하지 않는다.
- VDI 상위 25% 경계는 의료적 절대 임계값이 아니다.
- p-median·MCLP 결과는 현재 후보군 내부의 모델 비교다.
- 후보지는 현장조사 우선순위이며 확정 입지가 아니다.

## 8. 2차 독립심사 84점 후속 조치

확대된 심사 묶음에 대한 2차 평가에서는 이전의 확인 불가 항목 대부분이 코드로 확인됐고 판정이 **조건부 병합 가능 84/100**으로 상승했다. 정책 데이터 계약은 공개 가능한 수준으로 평가됐지만, 시민 화면이 일반응급실 병상 보고값을 `진료 가능` 또는 `수용 불가`로 번역하는 문제가 병합 전 P0로 지정됐다.

### 8.1 병상 보고 상태와 화면 문구

일반응급실 병상 상태의 내부 계약을 다음과 같이 변경했다.

| 원천 값 | 내부 상태 | 시민 표시 |
|---|---|---|
| `hvec=null` | `unknown` | 병상정보 미확인 |
| `hvec=0` | `reported-bed-zero` | 가용병상 0 보고 |
| `hvec>0` | `reported-bed-positive` | 가용병상 N개 보고 |

병상 양수는 특정 환자의 진료·수용 가능을 의미하지 않으며, 0도 병원 전체의 수용 상태를 확정하지 않는다. 데스크톱·모바일 지도 범례, 목록, 상세 배지와 정렬 근거에서 `진료 가능`, `수용 불가`, `당장 수용 가능` 표현을 제거했다.

기존 `isHospitalAvailable()`은 `hasReportedGeneralErBed()`로 교체했다. Tier 3 달빛어린이병원은 일반응급실 병상 수가 없더라도 자동 통과하던 예외를 제거해 `응급병상 보유만` 필터에서는 제외한다. 대신 `야간·휴일 소아진료` 조건에서 별도로 포함한다.

stale 상태 안내는 원인을 네트워크로 단정하지 않고 `원천 갱신 지연 또는 오류`로 표시한다. 실제 세부 원인은 인증 실패, 빈 응답, 오래된 자료, 초기화 미완료 또는 네트워크 오류일 수 있다.

### 8.2 스케줄러 장시간 작업 추적

APScheduler에 `lambda: asyncio.create_task(...)`를 등록하던 방식을 제거했다. 이제 `_run_target` coroutine과 대상 `args`를 직접 등록하므로 `max_instances=1`이 즉시 종료되는 Task 생성 함수가 아니라 실제 장시간 작업 전체를 추적한다. 오류 결과 로그에도 실행 소요시간을 기록한다.

테스트는 다음을 추가로 확인한다.

- 4개 작업 모두 `_run_target`을 직접 등록
- 작업별 대상 인자 일치
- 모든 작업의 `max_instances=1`
- Tier 3 + 병상 미확인 상태가 응급병상 필터에서 제외
- Tier 3가 야간·휴일 소아진료 필터에는 포함
- null·0·양수의 병상 보고 상태 분리

## 9. 최종 재심사 89점 후속 정리

최종 재심사에서는 병상 의미 표현과 스케줄러 실행 추적 문제가 실제 코드에서 해결된 것으로 확인돼 **병합 가능 89/100** 판정을 받았다. 남은 비차단 지적도 공개 화면과 저장소 증거의 정합성을 위해 다음과 같이 모두 보완했다.

### 9.1 지도 마커와 범례의 의미 일치

기존에는 `hvec=0`과 가용병상이 양수이지만 총병상 대비 비율이 낮은 상태가 모두 빨간색으로 표시될 수 있었다. 마커 색상 계약을 공통 함수로 중앙화해 다음처럼 구분했다.

| 색상 | 의미 |
|---|---|
| 빨강 | 가용병상 0 보고 |
| 주황 | 가용병상 양수 보고·비율 낮음 또는 중간 |
| 초록 | 가용병상 양수 보고 |
| 회색 | 병상정보 미확인 |
| 청록 | 야간·휴일 소아진료 기관 |

이 구분은 진료 가능 여부나 실제 수용 가능 여부를 뜻하지 않는다. `bedReportMarkerTone()` 단위 테스트로 0, 양수 저비율, 양수 고비율, 미확인, 달빛어린이병원 상태를 각각 검증한다.

### 9.2 공개 화면의 시간성 표현 정리

120초 폴링과 인메모리 캐시를 사용하는 조회 구조를 `실시간 병상`으로 과장하지 않도록 활성 시민 화면의 표현을 `최근 조회 병상정보`, `최근 조회 응급 병상정보`, `마지막 확인` 중심으로 변경했다. 원천 API의 조회값은 특정 환자의 진료·수용 가능을 보장하지 않는다는 안전 안내를 유지한다.

### 9.3 레거시 문서 구분

과거 구현과 시행착오를 담은 다음 문서는 파일을 이동해 기존 링크를 깨뜨리는 대신 문서 상단에 **현재 구현과 다른 과거 기록**임을 명시했다.

### 9.4 GitHub Actions 자동 검증

`.github/workflows/verify.yml`을 추가해 `main` 대상 Pull Request와 `main` push에서 다음 검증을 자동 실행하도록 했다.

### 9.5 교차 플랫폼 원본 해시 재현

Draft PR의 첫 Actions 실행에서 Windows 로컬 검증은 통과했지만 Linux runner의 분석 테스트가 인구 원본 파일 해시 불일치로 실패했다. 원인은 Git의 자동 줄바꿈 변환이었다. manifest는 검증 당시 CRLF 바이트의 SHA-256을 기록했지만 저장소에는 LF로 정규화돼 있어 Linux 체크아웃에서 다른 해시가 계산됐다.

인구 값과 manifest를 다시 계산하지 않고 검증 당시 원본 바이트를 보존하도록 `.gitattributes`에서 `data/raw/population/daegu_population_real.csv`를 binary로 지정하고 CRLF 바이트를 저장소에 고정했다. 이 파일은 내용 비교용 텍스트가 아니라 바이트 해시로 검증되는 원천 스냅샷이라는 계약을 명시한 것이다. 수정 후 분석 테스트 7개가 다시 통과했으며 Actions 전체 결과는 후속 실행에서 재검증한다.

### 9.6 CI 검증과 전체 재분석의 경계

해시 수정 후 두 번째 Actions 실행에서는 인구 원본 해시 검증, 분석 7개와 백엔드 29개 테스트가 통과했지만 `run_integrated_policy_pipeline.py --offline` 단계에서 중단됐다. 원인은 코드 오류가 아니라 `.gitignore`로 제외된 로컬 전용 `data/cache/kakao_road_eta_cache.json`이 깨끗한 runner에 존재하지 않기 때문이다. 이 캐시는 Kakao 도로 경로 5,100개의 수집 중간자료이며, 저장소 정본에는 최종 도로 행렬과 경로 결과 해시만 포함된다.

PR CI가 존재하지 않는 로컬 캐시를 전제로 전체 재분석 성공을 주장하지 않도록 검증 범위를 분리했다. GitHub Actions는 커밋된 25개 기관·5,100개 경로 정본의 계약을 테스트하고 단일 정책 릴리스를 재생성해 결정성을 확인한다. 7단계 전체 재분석은 로컬 캐시가 있거나 승인된 Kakao API 온라인 수집 환경에서 실행하며, 성공 후 검증본을 별도 PR로 승격한다. 이 구분은 CI 통과와 외부 API 재수집 성공을 혼동하지 않기 위한 운영 계약이다.

### 9.7 민감도 결과 해시의 운영체제 독립성

정책 릴리스 결정성 검사에서 민감도 JSON의 원시 파일 바이트 해시가 Windows CRLF와 Linux LF에 따라 달라지는 추가 문제가 확인됐다. JSON의 의미 내용은 같아도 줄바꿈만으로 릴리스 메타데이터가 달라지는 구조였다.

`sensitivity_sha256`을 원시 파일 바이트가 아니라 키 정렬과 고정 구분자로 직렬화한 JSON payload SHA-256으로 변경했다. 테스트도 같은 의미 기반 해시 계약을 검증하도록 수정했다. 분석 값은 변경하지 않으며, 정책 릴리스의 민감도 계보가 운영체제 줄바꿈 변환과 무관하게 동일해진다.

수정 후 Draft PR #12의 GitHub Actions 실행 `29647941237`에서 `Frontend tests and build`와 `Backend, analysis, and policy release`가 모두 통과했다. 이는 clean Linux runner에서 테스트, EDA 재생성, 정책 릴리스 재생성과 결정성 검사가 재현됨을 확인한 결과다.

## 10. 결론

이번 후속 개선으로 인구 기준월, 민감도 분석, 후보, 도로 경로, 최적화 결과가 더 긴 SHA-256 계보로 연결됐다. 좌표 보정과 실패 복원의 범위도 수치와 코드 기준으로 설명할 수 있게 됐다. 시민 화면의 역할 과장 문구와 EDA 해석 문제도 수정했다.

1차 기능·문서 개선은 PR #7과 #8을 통해, 병상 안전 표현과 스케줄러 실행 추적은 PR #9을 통해, 최종 공개 정합성 개선은 PR #10을 통해, 인구 원본 해시 재현성 수정은 PR #11을 통해 병합됐다. 로컬 경로 캐시가 필요한 전체 재분석과 clean runner의 정책 정본 검증을 분리하는 수정은 [Draft PR #12](https://github.com/ssg-sak/golden-project/pull/12)에서 검토하며 사용자 확인 전 병합하지 않는다. 코드와 자동 검증이 강화됐더라도 외부 API 조회값의 지연, 병상 보고값과 실제 수용 가능 여부의 차이, 정책 모델의 한계는 계속 명시한다.
