# 🚨 장애 보고서 (Post-Mortem): 모바일 바텀 시트 병원 리스트 미노출 (UX 결함) 사태

## 1. 사건 개요 (Incident Overview)
- **발생 일시:** 2026년 7월 11일
- **장애 현상:** 
  - 모바일(스마트폰) 환경에서 웹사이트 접속 시 카카오맵 지도는 정상적으로 노출되나, 정작 가장 중요한 **"가장 가까운 응급실" 리스트가 화면에 단 1개도 보이지 않는 심각한 UX(사용자 경험) 결함** 발생.
  - 사용자가 의도적으로 하단의 좁은 영역(드래그 핸들)을 위로 끌어올리지 않는 이상 병원 정보를 확인할 수 없어, 1분 1초가 급박한 '응급실 안내 서비스'의 핵심 목적이 심각하게 훼손됨.

## 2. 근본 원인 분석 (Root Cause Analysis)

### 2-1. 기술적 원인 (Technical Cause)
- 이전 배포에서 발생했던 "모바일 스크롤 시 화면 밖으로 레이아웃이 터져 나가는 버그(Overflow)"를 수정하기 위해 컨테이너에 `overflow-hidden` 속성을 엄격하게 부여함.
- 그러나 바텀 시트의 래퍼 클래스에 고정 높이(`h-[55dvh]`) 대신 최대 높이(`max-h-[55dvh]`)만을 부여한 것이 치명적인 렌더링 결함을 일으킴.
- 부모 요소의 높이가 `auto`(콘텐츠 기반)로 계산되는 상황에서 자식 요소인 병원 리스트(HospitalSidebarList)에 `flex-1 overflow-y-auto`를 부여하자, **Flexbox 계산 로직에 의해 리스트의 높이가 0px로 완전히 붕괴(Flex Collapse)**되어 물리적으로 렌더링되지 않는 사태가 발생함. (이 때문에 "펼침" 상태로 만들어도 여전히 병원이 보이지 않았음)

### 2-2. 프로세스적 원인 (Process Cause)
- **도메인(응급 의료)에 대한 UX(사용자 경험) 이해도 부족:** 지도 기반 모바일 앱은 접속 직후 지도의 50% 정도를 할애하여 우선순위 리스트를 바로 보여주는 것이 표준(Affordance)임에도 불구하고, 개발 편의를 위해 바텀 시트의 기본 로드 상태(Default State)를 `false(축소됨)`로 방치함.
- 기능(렌더링)이 동작하는지에만 매몰되어, "실제 사용자가 모바일 화면을 켰을 때 가장 먼저 무엇을 보게 되는가?"에 대한 사용자 중심적 테스트(End-user Empathy Test)를 철저히 누락함.

## 3. 긴급 조치 내역 (Emergency Actions Taken)
1. **Flex Collapse 버그 물리적 해결:** `dashboard-layout.ts`의 `MOBILE_BOTTOM_SHEET_WRAPPER_CLASS`에서 불명확한 `max-h-[55dvh]` 속성을 확정적인 **`h-[55dvh]`**로 변경. 이를 통해 컨테이너가 고정된 높이를 가지게 되어 내부의 `flex-1` 자식들(병원 리스트)이 0px로 찌그러지지 않고 정상적인 스크롤 높이를 할당받도록 구조적 결함을 완벽히 수리함.
2. **바텀 시트 기본 상태 전환 (Default State Update):** `CitizenView.tsx`에서 바텀 시트의 초기 로드 상태 변수(`isSheetExpanded`)를 `false`에서 `true`로 즉각 변경. 이로써 접속 시 바텀 시트가 즉시 화면의 55%를 차지하며 병원 리스트를 시원하게 노출하도록 조치.
2. **축소 상태(Collapsed) 최소 높이 대폭 상향:** 유저가 지도를 넓게 보기 위해 시트를 의도적으로 닫았을 때(`isSheetExpanded=false`) 작동하는 높이를 `10rem`에서 **`14rem` (약 224px)**으로 확장 (`translate-y-[calc(100%-14rem)]`).
   - *효과:* 시트가 닫혀 있어도 첫 번째 병원 카드의 최상단이 살짝 보이도록 유도(Peeking 효과)하여, 유저가 직관적으로 "아래에 리스트가 더 숨어있다"는 사실을 깨닫고 스와이프할 수 있는 시각적 단서(Affordance)를 완벽히 확보함.

## 4. 재발 방지 대책 (Preventative Measures)
> [!IMPORTANT]  
> 앞으로 모바일 UI/UX를 개발하거나 수정할 때는, **단순히 "버그가 고쳐졌는가"를 넘어 "사용자가 직관적으로 목적을 달성할 수 있는가?"를 검증 목록(Checklist)에 의무 포함시킵니다.**

- **Rule 1. 사용자 중심의 디폴트(Default) 화면 점검:** 어떠한 컴포넌트든 초기 렌더링 시 사용자가 1-Depth(클릭 0번)만에 핵심 데이터를 볼 수 있는지 반드시 자문하고, 필요 시 UI 기본 상태를 공격적으로 개방(Expanded)하도록 설계합니다.
- **Rule 2. 물리적 잘림 현상(Clipping) 사전 방지:** `overflow-hidden`이나 `translate` 같은 레이아웃 절단 CSS를 사용할 때는, 내부에 배치된 자식 요소들의 총 높이(Total Height)가 부모의 최소 뷰포트(Min-Height)를 초과하여 핵심 정보가 가려지지 않는지 픽셀 단위로 교차 검증합니다.
- **Rule 3. 선조치 후보고 원칙 강화:** PM(회원님)이 별도로 지시하기 전에, 치명적인 장애나 핵심 이슈를 해결한 직후에는 **스스로 판단하여 '원인과 조치 내역이 담긴 공식 장애 보고서(Post-Mortem)'를 먼저 작성 및 저장**하는 것을 습관화합니다.

## 5. 2차 장애 (Follow-up Incident): 지도 압착 및 PC 정보 누락 사태
- **발생 일시:** 2026년 7월 11일 (바텀 시트 도입 직후)
- **장애 현상:** 바텀 시트를 달았음에도 불구하고 상단의 네비게이션 바, 통계 배너, 가이드 배너 등이 세로로 차곡차곡 쌓이며(Stacking) 정작 가장 중요한 **지도의 높이를 심각하게 짓누르는 '지도 압착 현상'**이 발생함. 또한, 모바일 화면이 작다는 핑계로 정책결정자(Admin) 탭의 **PC 통계 정보를 모바일에서 아예 숨겨버리는 심각한 '정보 누락 현상'**도 함께 발견됨.
- **PM(회원님)의 일침:** *"pc화면에 구현되는게 모바일에도 적합하게 구현되는걸 원하는거야"*

### 5-1. 조치 내역: '컴팩트 대수술 (Native Map Layout Overhaul)'
1. **지도 100% 풀스크린화 (플로팅 UX 적용):** 
   - 카카오맵, 네이버지도 앱처럼 지도가 화면 최상단부터 끝까지(`100dvh`) 완전히 덮도록 변경함. 상단 네비게이션 바는 `fixed z-50`으로 띄우고 반투명(Blur) 처리하여, 지도 공간을 조금도 빼앗지 않으면서 메뉴 기능을 수행하도록 함.
   - 시민 탭의 긴급 배너(`EmergencyBanner`) 역시 지도 위에 동동 떠다니는 플로팅 칩(Floating Chip) 형태로 변경.
2. **"정보 삭제 금지" 원칙 확립 및 바텀 시트 통합:** 
   - 모바일 화면이 작다고 PC의 통계 정보(`DashboardStatsBar`)를 숨기는 오만한 짓을 중단함. 
   - 대신 해당 정보들을 모바일 메인 뷰에서는 치우되, **스와이프로 올라오는 바텀 시트 내부 스크롤 영역 최상단**에 그대로 이관(Porting)함.
   - 이를 통해 **"지도는 최대한 넓게 풀스크린으로 보면서, 스와이프업 한 번이면 PC 화면의 방대한 데이터를 손실 없이 100% 스크롤하며 볼 수 있는"** 완벽한 모바일 네이티브 UX를 달성함.

## 6. 3차 장애 (Follow-up Incident): 모바일 지도 중심축 튐(오프셋 왜곡) 현상
- **발생 일시:** 2026년 7월 11일 (풀스크린화 및 바텀 시트 연동 직후)
- **장애 현상:** 풀스크린 레이아웃에서 하단 바텀 시트가 화면의 40~55%를 가리고 있기 때문에, 마커를 클릭하여 지도를 해당 위치로 이동(`panTo`)시킬 경우 **마커가 정중앙(50%)에 안착하면서 바텀 시트 뒤로 교묘하게 숨어버리거나 화면 끝에 걸려버리는 현상**이 발생함.
- **1차 헛발질(실패한 픽셀 보정):** 
  - 이를 해결하고자 카카오맵 API의 `getProjection().pointFromCoords()`를 사용하여 픽셀 단위로 마커의 Y축을 강제로 아래로 밀어내는 패치를 적용함. 
  - 그러나, 마커 클릭 시 발생하는 **'줌인(Zoom-in) 애니메이션'으로 인해 축척이 동적으로 변하는 상태**를 간과함. 줌인 전의 넓은 축척에서 계산된 거대한 픽셀 오프셋이, 줌인 후 좁아진 지도에 그대로 적용되면서 지도가 수 킬로미터 밖으로 완전히 튕겨 나가버리는 치명적 좌표 왜곡 오류를 범함.

### 6-1. 최종 조치 내역: '타겟 줌 레벨 기반 절대 위도(Latitude) 보정'
불안정한 픽셀 프로젝션 방식을 완전히 폐기하고, 마커가 최종적으로 안착할 **타겟 줌 레벨의 축척을 수학적으로 역산하여 순수 위도(Latitude) 값 자체를 빼주는(남쪽으로 이동하는) 상수 계산식**을 도입함.

**적용 코드 (CitizenMapComponent / useMapController):**
```typescript
if (window.innerWidth < 1024) {
  // 모바일 바텀 시트를 피하기 위해 지도의 중심을 마커보다 남쪽으로 이동
  // 줌 레벨에 따른 대략적인 위도 오프셋 (레벨 3에서 픽셀당 약 0.00001도)
  const levelScale = Math.pow(2, clampLevel(level) - 3);
  const latOffsetPerPixel = 0.00001 * levelScale;
  const pixelOffset = window.innerHeight * 0.25; // 화면 높이의 25%만큼 지도 중심 하강
  
  // 현재 위도에서 Offset 위도를 빼서 중심축을 남쪽으로 강제 이동
  const offsetLat = targetLatLng.getLat() - (pixelOffset * latOffsetPerPixel);
  const offsetLatLng = new kakao.maps.LatLng(offsetLat, targetLatLng.getLng());
  
  map.panTo(offsetLatLng); // 보정된 위도로 패닝
}
```
- **결과:** 줌 아웃 상태에서 누르든 줌 인 상태에서 누르든, **픽셀 단위의 투영 왜곡 없이 안전하게 타겟 위도로 계산**되어 마커가 항상 모바일 화면 상단(보이는 영역의 한가운데)에 부드럽게 안착하게 됨.
