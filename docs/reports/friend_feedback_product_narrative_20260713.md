# 친구의 한마디를 제품 판단으로 바꾼 과정

## 한 줄 서사

“가까운 응급실”을 보여주는 데서 멈추지 않고, 실제 사용자의 질문을 계기로 병상 여유와 이동시간을 함께 판단하는 추천으로 발전시켰다.

## 1. 출발점: 기능은 있었지만 판단은 사용자 몫이었다

기존 서비스에는 현재 위치, 응급실 병상 수, 차량 이동시간, 카카오 길찾기가 있었다. 각각은 동작했지만 목록은 주로 거리나 ETA 순이었다. 가까운 병원의 병상이 없을 때 사용자는 다시 여러 병원의 숫자를 비교해야 했다.

친구의 피드백은 기술 기능이 아니라 실제 상황의 언어로 들어왔다.

- 혼잡 여부를 빨강·노랑·초록으로 바로 보고 싶다.
- 가까운 곳에 자리가 없으면 조금 멀어도 여유 있는 곳을 먼저 추천해 달라.
- 구급차로 이동할 때 가장 빠른 경로를 보고 싶다.

이 질문 덕분에 “데이터를 표시하는 서비스”와 “판단을 돕는 서비스”의 차이가 드러났다.

## 2. 그대로 구현하지 않고 데이터의 한계를 먼저 확인했다

피드백의 표현을 그대로 기능명으로 옮기지 않았다.

- 가용 병상으로는 실제 대기시간을 알 수 없으므로 `대기 없음`이라고 쓰지 않았다.
- 병상 여유가 있어도 의료진·환자 상태에 따라 수용이 거절될 수 있으므로 `수용 확정`이라고 쓰지 않았다.
- 공개 지도 API의 일반 차량 경로로는 역주행·신호 특례·119 관제를 재현할 수 없으므로 `응급차 전용 경로`를 만들지 않았다.

대신 데이터가 책임 있게 말할 수 있는 범위에서 `병상 여유 우선 추천`과 `일반 차량 길찾기`를 제공했다.

## 3. 제품 규칙으로 번역했다

친구의 말은 다음 추천 정책으로 바뀌었다.

```text
원활 → 지연 → 혼잡하지만 가용 병상 있음 → 여유 없음 → 확인 필요
같은 상태라면 차량 ETA → ETA가 없으면 직선거리
```

최상위 병원에는 추천 배지뿐 아니라 선정 근거와 “실제 수용 여부는 출발 전 확인” 안내를 함께 표시했다. 사용자는 결과뿐 아니라 추천 이유와 한계도 볼 수 있다.

## 4. 배포 현실까지 포함해 완성했다

공개 데모는 GitHub Pages 정적 배포다. 서버가 없으면 Kakao REST API 기반 ETA를 계산할 수 있다. 그래서 ETA를 필수 조건으로 만들지 않았다.

- GitHub Pages: 병상 상태와 직선거리로 추천
- FastAPI 연결 환경: 병상 상태와 차량 ETA로 추천
- ETA 장애: 직선거리로 자동 대체

동일한 기능이 데모 환경에서도 사라지지 않게 설계하면서 배포 제약을 제품 규칙에 포함했다.

## 5. 구현과 검증

추천 정책은 React 화면에서 분리한 공용 비교 함수로 구현했다. 데스크톱, 모바일, 지도 없는 목록이 동일한 규칙을 사용한다.

- 단위 테스트 14개 통과
- TypeScript 타입 검사 통과
- ESLint 통과
- GitHub Pages 데모 빌드 통과
- 로컬 FastAPI `/api/hospitals/runtime-config`, `/api/hospitals` 응답 확인
- 로컬 Vite 프런트엔드 HTTP 200 확인

## 6. 이 경험이 보여주는 역량

이 사례의 핵심은 친구의 아이디어를 많이 구현했다는 데 있지 않다.

1. 비개발자의 현장 언어에서 실제 사용자 문제를 발견했다.
2. 데이터가 말할 수 있는 범위와 말할 수 없는 범위를 구분했다.
3. 위험한 요구는 거절하는 대신 목적을 살리는 안전한 대안을 만들었다.
4. 정적 배포와 API 장애를 고려해 기능이 유지되는 폴백을 설계했다.
5. 판단 기준을 테스트와 문서로 남겨 다시 설명하고 검증할 수 있게 했다.

## 포트폴리오용 짧은 답변

“처음에는 현재 위치에서 가까운 응급실과 병상 수를 보여주는 기능이었습니다. 친구가 ‘가까운 곳에 자리가 없으면 조금 멀어도 여유 있는 병원을 먼저 보여주는 게 낫지 않겠냐’고 피드백했습니다. 확인해 보니 기존 목록은 정보를 제공하지만 사용자의 비교 부담을 줄이지 못하고 있었습니다. 그래서 병상 여유를 먼저, 같은 상태에서는 차량 ETA와 거리를 비교하는 추천 정책을 만들었습니다. 다만 병상 수만으로 실제 대기 없음이나 수용 확정을 말할 수 없고, 일반 지도 API로 응급차 특수경로도 만들 수 없었습니다. 이 한계를 화면에 명시하고 119 관제 안내로 대체했습니다. 정적 GitHub Pages에서는 거리 폴백으로 동작하게 해 배포 환경에서도 기능을 유지했고, 추천 순서와 안전 문구를 단위 테스트로 고정했습니다.”
