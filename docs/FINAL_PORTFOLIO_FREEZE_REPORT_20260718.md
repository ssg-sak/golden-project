## 3. 해결한 주요 문제

### 3.1 병상 조회값의 의료적 의미 과장

과거 화면은 병상 수가 양수이면 `진료 가능`, 0이면 `수용 불가`처럼 해석될 여지가 있었다. 그러나 공개 API의 병상 보고값만으로 특정 환자의 진료 또는 수용 가능 여부를 확정할 수 없다.

최종 상태 계약은 다음과 같다.

| 원천 값 | 내부 상태 | 시민 표시 |
|---|---|---|
| `hvec=null` | `unknown` | 병상정보 미확인 |
| `hvec=0` | `reported-bed-zero` | 가용병상 0 보고 |
| `hvec>0` | `reported-bed-positive` | 가용병상 N개 보고 |

지도·목록·상세·필터에서 진료·수용 가능을 단정하는 문구를 제거했다. 달빛어린이병원은 일반응급실 병상 필터에 자동 포함하지 않고 야간·휴일 소아진료 조건으로 구분한다.

### 3.2 지도 마커와 범례 불일치

가용병상 0과 양수이지만 총병상 대비 비율이 낮은 상태가 모두 빨간색으로 표시되던 문제를 수정했다. 마커 색상은 공통 상태 함수로 중앙화했고 데스크톱·모바일 범례와 같은 계약을 사용한다.

| 색상 | 의미 |
|---|---|
| 빨강 | 가용병상 0 보고 |
| 주황 | 가용병상 양수 보고·비율 낮음 또는 중간 |
| 초록 | 가용병상 양수 보고 |
| 회색 | 병상정보 미확인 |
| 청록 | 야간·휴일 소아진료 기관 |

색상은 공식 API의 마지막 조회 상태를 설명할 뿐, 환자별 진료 가능 여부나 실제 수용 결정을 뜻하지 않는다.

### 3.3 공개 화면과 검색 메타의 시간성 표현

120초 폴링과 인메모리 캐시 구조를 `실시간 병상`으로 단정하지 않도록 활성 시민 화면을 `최근 조회 병상정보`, `최근 조회 응급 병상정보`, `마지막 확인` 중심으로 정리했다.

검색 결과와 링크 미리보기에서도 같은 원칙을 유지하도록 `frontend/index.html`의 설명을 다음처럼 수정했다.

### 3.5 Windows와 Linux의 해시 재현성

Git의 줄바꿈 변환 때문에 동일한 의미의 파일이 Windows에서는 CRLF, Linux에서는 LF로 체크아웃되면서 원시 바이트 SHA-256이 달라지는 문제가 발견됐다.

조치는 파일 성격에 따라 구분했다.

1. 인구 원본 CSV는 manifest가 원본 바이트를 검증하므로 `.gitattributes`에서 binary 스냅샷으로 고정했다.
2. 인구 manifest JSON과 민감도 JSON은 의미 내용의 계보를 검증하므로 canonical JSON payload 해시를 사용한다.

canonical JSON은 키 정렬, 고정 구분자, UTF-8 직렬화를 사용한다. 현재 인구 manifest의 canonical 해시는 다음과 같다.

```text
7391558cc20d33de823f9e962ad88ee0ee1e20a6dd32fdd8d980f7698a9fabec
```

이 변경은 인구, VDI, 후보, 도로 경로 또는 최적화 값을 바꾸지 않는다. 운영체제에 따라 메타데이터 해시만 달라지던 문제를 제거한다.

## 6. CI와 전체 7단계 재분석의 경계

GitHub Actions는 커밋된 정책 정본의 계약과 재생성 결정성을 검증한다. 그러나 `.gitignore` 대상인 로컬 Kakao 도로 경로 캐시 5,100건을 보유하지 않으므로 외부 경로를 처음부터 재수집하는 전체 분석 성공까지 주장하지 않는다.

전체 재분석은 다음 환경에서 수행한다.

- 검증된 로컬 경로 캐시가 있는 개발 환경
- 또는 승인된 Kakao API 키와 호출 한도가 있는 온라인 수집 환경

전체 재분석 절차는 다음과 같다.

1. 행정동·인구·기관 공간 전처리
2. 후보 민감도 분석
3. 안정 후보 정본 생성
4. 후보 추적 데이터 생성
5. 실제 도로망 행렬과 입지 최적화
6. 후보 시각화와 최종 계약 검증
7. 단일 정책 릴리스 생성

실패하면 기존 검증본을 복원하며, 성공 결과도 자동으로 공개하지 않고 검토용 PR에서 검증한 뒤 공개본으로 승격한다.

## 10. 남은 한계

다음은 미해결 버그가 아니라 서비스와 분석이 계속 밝혀야 하는 운영·연구 한계다.

- CI는 커밋된 5,100개 경로 정본의 무결성과 결정성을 검증하며 Kakao API에서 매번 5,100개 경로를 새로 수집하는 온라인 전체 재분석까지 증명하지 않는다.
- Render의 관리자 토큰 등록, 단일 인스턴스 운영, 외부 병상 API의 실제 지연·실패 빈도는 저장소만으로 완전히 검증할 수 없다.
- 일반 차량 도로 ETA는 실제 119 구급차 이송시간이 아니다.
- 병상 보고값은 특정 환자의 실제 수용 가능 여부를 확정하지 않는다.
- VDI는 보호 필요 인구와 접근 부담을 결합한 비교 지표이며 의료적 절대 위험 기준이 아니다.
- 위험 상위 구간과 임계값은 정책 검토 우선순위이지 진단 또는 확정 결론이 아니다.
- 후보지와 자원 보강 시나리오는 현장조사, 토지·예산·인력 검토 전의 분석 결과다.
- 외부 API의 지연, 장애, 호출 한도와 원천 데이터 품질 문제는 완전히 제거할 수 없다.

- 원천 데이터 갱신과 새 분석 버전 발행
- 보안 업데이트
- 접근성·문서 가독성 개선
- 배포 장애 복구

새 기능 제안은 현재 검증본에 즉시 덧붙이지 않고 별도 이슈와 브랜치에서 필요성, 데이터 근거, 안전 영향, 검증 방법을 먼저 정의한다.

## 12. 포트폴리오 설명 핵심

대구 골든타임은 가까운 응급의료기관을 찾는 시민 서비스와 대구 행정동별 의료 접근성 격차를 분석하는 골든 거버넌스 엔진을 하나의 웹서비스로 구성한 개인 프로젝트다.

프로젝트의 핵심 성과는 화면 기능의 수가 아니라 다음에 있다.

- 불완전한 공공 API 값을 의료적 판단으로 과장하지 않는 상태 설계
- 150개 행정동·25개 기관·9개 후보·5,100개 도로 경로의 단일 정책 정본
- 인구 수요, 의료 공급, 도로 접근성, 취약성 지수, 후보·최적화 결과의 계보 연결
- 원천 갱신 실패 시 기존 검증본을 보존하는 승격·복원 구조
- Windows와 Linux에서 동일한 의미 해시를 만드는 재현성 보강
- 로컬 전체 재분석과 GitHub CI 검증의 책임을 정직하게 분리한 운영 계약

이제 프로젝트의 다음 단계는 기능 추가가 아니라 README와 실제 화면을 중심으로 문제 정의, 안전 설계, 분석 방법, 검증 실패에서 배운 점을 면접에서 명확히 설명하는 것이다.
