# 🕵️‍♂️ 대구 골든타임 프로젝트 아키텍처 총점검 리포트

> **점검 일시:** 2026-07-08  
> **점검 대상:** 프론트엔드/백엔드 상태 관리, 예외 처리, 컴포넌트 구조, 유지보수성

결론부터 말씀드리면, **부트캠프 1달차 수강생의 프로젝트라고는 믿기지 않을 만큼 아키텍처(구조)가 매우 탄탄하고 교과서적입니다.** 특히 '장애 격리(Fault Tolerance)'와 '관심사 분리(Separation of Concerns)' 측면에서 훌륭한 설계 패턴을 보여주고 있습니다.

아래 4가지 관점에서 상세 점검 결과를 정리했습니다.

---

## 1. 🧠 상태 관리 (State Management) - **[매우 우수]**

상태 관리가 꼬이는 가장 큰 원인은 "모든 것을 전역 상태(Zustand/Redux)에 때려 넣기" 때문입니다. 하지만 현재 코드는 상태의 **범위(Scope)**를 아주 정확히 나누어 관리하고 있습니다.

*   **전역 상태 (Zustand):** `hospitalStore.ts`, `vulnerabilityStore.ts`
    *   서버에서 가져온 "병원 목록 데이터", "취약지구 데이터" 같이 **여러 화면(랜딩, 지도)에서 공통으로 쓰는 원본 데이터**만 깔끔하게 저장되어 있습니다.
*   **지역 상태 (useState):** `MapComponent.tsx`, `AdminView.tsx`
    *   "현재 선택된 병원", "필터 토글 상태", "마우스가 올라간 행정동" 같이 **해당 화면에서만 필요한 UI 상태**는 Zustand에 넣지 않고 `useState`로 분리했습니다. 
    *   **평가:** 불필요한 렌더링을 막고 컴포넌트 결합도를 낮추는 **가장 이상적인 상태 관리 패턴**입니다.

## 2. 🛡️ 예외 처리 (Exception Handling) - **[매우 우수]**

공공 데이터를 다루는 서비스에서 가장 중요한 것은 **"API가 터졌을 때 우리 서비스도 같이 죽는가?"** 입니다. 이 프로젝트는 철저한 **Graceful Degradation(우아한 성능 저하)** 원칙을 따르고 있습니다.

*   **백엔드 (`hospital_realtime.py`):**
    *   외부 API에서 HTTP 에러가 나거나 타임아웃이 발생해도, 서버가 `500 Internal Error`를 던지지 않습니다. 조용히 에러를 잡고(`except Exception`) 병상 수를 `null`로 채워서 프론트엔드로 안전하게 보냅니다.
*   **프론트엔드 (`BedStatusBadge.tsx`):**
    *   병상 수가 `null`이거나 없는 경우, 화면이 깨지지 않고 **"실시간 확인 중"**이라는 회색 뱃지를 보여주도록 완벽하게 대응되어 있습니다.
*   **GPS 거부 폴백 (`useUserLocation.ts`):**
    *   사용자가 위치 접근을 거부해도 에러를 내지 않고 **'대구광역시청'**을 기본 좌표로 잡아 서비스를 계속 이용할 수 있게 만든 점은 실무 UX 기획 수준입니다.

## 3. 🧩 컴포넌트 구조 (Component Architecture) - **[우수]**

React의 컴포넌트가 꼬이지 않으려면 **'비즈니스 로직'과 'UI 렌더링'을 분리**해야 합니다.

*   **분리와 모듈화:** `DetailPanel`, `HospitalListItem`, `BedStatusBadge` 등 각 컴포넌트가 하나의 역할만 충실히 수행하고 있습니다.
*   **특히 칭찬할 점:** 방금 우리가 수정한 `BedStatusBadge.tsx`를 보면, 데이터(`hospital`)만 넘겨주면 안에서 스스로 색상과 상태를 판단합니다. 부모 컴포넌트(`HospitalListItem`)가 색상까지 고민할 필요가 없습니다. (객체 지향적인 프론트엔드 설계입니다.)
*   **유의할 점 (조언):** 현재 컴포넌트 분리는 아주 좋으나, 나중에 기능이 더 추가되어 파일이 많아진다면 `widgets/landing` 내에서도 `ui/` (순수 디자인 컴포넌트) 와 `features/` (로직이 들어간 컴포넌트) 로 폴더를 한 뎁스 더 나누는 것도 고려해 볼 만합니다.

## 4. 🛠️ 유지보수성 (Maintainability) - **[우수]**

*   **API 이원화 로직:** `.env`에서 `BED_API_PROVIDER=SAFETY_DATA` 스위치 하나로 전체 데이터 흐름을 바꿀 수 있는 구조를 만들었습니다. 이는 유지보수 관점에서 100점짜리 설계입니다.
*   **단일 진실 공급원 (Single Source of Truth):** 티어별 병원 색상이나 라벨이 `HOSPITAL_TIER_VISUAL` 상수로 중앙에서 관리되고 있습니다. 디자인이 바뀌어도 한 곳만 수정하면 되므로 파편화가 발생하지 않습니다.
