# Golden Data Lab 별도 저장소 분리 계획

## 문서 상태

- 대상 폴더: golden-data-lab/
- 현재 저장소: ssg-sak/golden-project
- 기준 브랜치: main
- 상태: 계획 초안
- 이 문서에서는 새 저장소 생성, 파일 이동, 원본 삭제를 수행하지 않는다.
- 실제 분리는 저장소 소유자의 별도 승인 후 독립 작업으로 진행한다.

## 현재 확인된 범위

GitHub 원격 main에서 다음 파일을 확인했다.

- golden-data-lab/README.md
- golden-data-lab/ROADMAP.md
- golden-data-lab/01_sql_public_health_analysis/README.md
- golden-data-lab/notebooks/.gitkeep

README와 ROADMAP에는 SQL, Python EDA, Power BI, 통합 보고서를 단계적으로 학습·구현하는 별도 데이터 분석 프로젝트 구상이 기록돼 있다. 원격 코드 검색은 빈 폴더와 검색어가 없는 바이너리·데이터를 완전하게 열거하지 못할 수 있으므로, 실제 분리 전 Git 트리 전체 인벤토리를 한 번 더 확인해야 한다.

## 1. 별도 저장소로 분리하는 이유

대구 골든타임은 시민용 응급의료 서비스와 골든 거버넌스 정책 엔진을 배포하는 제품 저장소다. 반면 Golden Data Lab은 SQL·Python EDA·Power BI·보고서 작성 역량을 단계적으로 확장하는 학습·분석 작업공간이다.

분리하면 다음 경계가 선명해진다.

- 제품 저장소: React, FastAPI, 운영 데이터 계약, 정책 엔진, 테스트, 배포
- 데이터 랩 저장소: 질문 중심 SQL, 탐색적 분석, 실험 노트북, BI 대시보드, 학습 보고서
- 제품 릴리스와 분석 실험의 커밋·이슈·의존성 분리
- 공개 가능한 샘플과 재배포 제한 원천 데이터의 관리 분리
- 채용 포트폴리오에서 웹서비스와 분석 역량을 각각 설명

분리는 코드 품질 문제를 숨기기 위한 조치가 아니라 서로 다른 수명주기와 독자를 가진 프로젝트를 명확히 구분하기 위한 조치다.

## 2. 권장 저장소 이름

권장 이름: **daegu-golden-data-lab**

이유:

- golden-project와 연결 관계가 보이면서도 독립 분석 저장소임을 알 수 있다.
- 지역·도메인·역할이 이름에 포함돼 검색성이 좋다.
- 향후 다른 주제의 데이터 랩과 구분하기 쉽다.

대안:

- golden-data-lab: 현재 폴더명과 동일해 단순하지만 범위가 모호하다.
- daegu-emergency-access-data-lab: 분석 주제가 가장 명확하지만 이름이 길다.

## 3. 이동할 파일

1차 이동 대상은 golden-data-lab/ 아래에서 실제 Git 추적이 확인된 파일 전체다.

현재 확인된 대상:

- README.md
- ROADMAP.md
- 01_sql_public_health_analysis/README.md
- notebooks/.gitkeep

분리 직전 추가 확인 대상:

- SQL 쿼리와 스키마
- Python EDA 코드와 노트북
- Power BI 템플릿·내보내기 파일
- 분석 결과 CSV·JSON·이미지
- 데이터 사전·출처·라이선스 문서
- 테스트 또는 재현 스크립트

새 저장소 루트로 옮길 때는 golden-data-lab/ 접두 경로를 제거해 README.md와 ROADMAP.md가 저장소 최상단에 오도록 한다.

## 4. 제외할 파일

다음 항목은 자동으로 복사하지 않는다.

- 실제 .env와 API 키·토큰·자격 증명
- data/cache/, tmp/, 로그, 테스트 결과, 노트북 체크포인트
- node_modules/, 가상환경, 빌드 산출물
- 데이터베이스 비밀번호나 개인 식별정보가 포함된 파일
- 재배포 권한이 확인되지 않은 원본 공공데이터
- golden-project의 frontend/, backend/, ai-model/ 전체 복사본
- GitHub Pages·Render 등 제품 배포 설정
- 제품 저장소의 실시간 병상 캐시·운영 DB
- 동일 산출물의 중복 백업

제품 분석 코드를 데이터 랩에서도 사용해야 하면 전체 폴더를 복제하지 말고 다음 중 하나를 선택한다.

1. 버전이 표시된 공개 산출물만 가져온다.
2. 재사용 가능한 최소 모듈을 별도 승인 후 패키지화한다.
3. 원본 저장소의 특정 릴리스·커밋을 데이터 출처로 링크한다.

## 5. Git 기록 보존 여부

권장: **golden-data-lab 경로의 Git 기록을 보존한다.**

이유:

- 분석 질문과 구조가 어떻게 발전했는지 확인할 수 있다.
- 데이터·방법론 변경의 근거를 추적할 수 있다.
- 작성자와 기준 시점의 이력을 유지할 수 있다.

권장 절차는 임시 복제본에서 git-filter-repo 또는 동등한 검증된 도구로 golden-data-lab/ 경로만 추출하고, 접두 폴더를 새 저장소 루트로 바꾸는 방식이다. 원본 저장소나 사용자의 주 작업 폴더에서 직접 파괴적 히스토리 재작성은 하지 않는다.

기록 보존이 필요 없다고 별도 승인한 경우에만 GitHub API로 현재 파일 스냅샷을 새 저장소 첫 커밋으로 만들 수 있다. 이 경우 이전 이력이 사라진다는 점을 새 저장소 README에 기록한다.

## 7. 기존 저장소에서 제거할 시점

golden-project에서 golden-data-lab/을 제거하는 PR은 다음 조건을 모두 만족한 뒤 별도로 만든다.

- 새 저장소 생성에 대한 사용자 승인
- 전체 파일 인벤토리와 비밀정보 점검 완료
- 새 저장소로 파일·Git 기록 이전 완료
- 새 저장소 기본 브랜치 보호와 공개 범위 확인
- README의 데이터 출처·라이선스·실행 방법 작성
- 주요 링크와 샘플 데이터 검증
- golden-project 코드·문서·워크플로의 경로 참조 검색 완료
- 두 저장소의 상호 링크 PR 준비
- 사용자가 새 저장소 내용을 직접 검토

제거 PR에서는 폴더 삭제와 링크 수정만 다루고 제품 소스 리팩터링을 섞지 않는다.

## 8. 두 저장소를 연결하는 방법

초기 권장 방식은 서브모듈이 아니라 문서 링크다.

- golden-project README의 상세 분석·학습 섹션에서 새 저장소 링크
- 새 저장소 README에서 배포 서비스와 golden-project 링크
- 분석 산출물에 source repository, source commit, data reference date 기록
- 필요하면 같은 이름의 GitHub Topic 사용
- 제품에서 소비하는 산출물은 버전·생성일·스키마를 명시

서브모듈은 배포·온보딩 복잡도를 높이므로 실제 자동 동기화 요구가 생기기 전에는 사용하지 않는다.

## 9. 분리 후 검증 절차

### 파일·기록

- 원본 인벤토리와 새 저장소 파일 목록 비교
- 빈 폴더 유지용 .gitkeep 확인
- golden-data-lab 관련 커밋 기록 보존 확인
- 대용량·바이너리 파일과 Git LFS 필요 여부 확인

### 보안·데이터

- 실제 .env와 비밀값이 없는지 재검사
- 공개 데이터 키와 서버용 비밀키 구분
- 개인정보·민감정보 포함 여부 확인
- 각 데이터의 출처·라이선스·재배포 가능 여부 확인
- 원본 데이터 미포함 시 재수집 방법 또는 제외 이유 기록

### 재현성

- 깨끗한 환경에서 README 순서대로 실행
- SQL 스키마와 샘플 쿼리 실행
- Python 의존성 설치와 EDA 실행
- 난수 시드와 데이터 기준일 확인
- 생성 결과의 스키마·행 수·핵심 지표 대조
- Power BI 파일은 데이터 원본 경로와 새로 고침 절차 확인

### 연결·배포 영향

- golden-project에서 golden-data-lab 경로 참조가 남지 않았는지 확인
- 양쪽 README 링크 확인
- golden-project 빌드·테스트·GitHub Pages 배포 확인
- 시민·정책 서비스가 새 저장소 없이도 정상 동작하는지 확인

## 10. 문제가 생겼을 때 되돌리는 방법

1. golden-project에서 폴더를 제거하는 PR은 즉시 병합하지 않고 Draft 상태에서 검증한다.
2. 제거 직전 main 커밋 SHA와 폴더 트리를 기록한다.
3. 문제가 발견되면 제거 PR을 닫거나, 병합 후라면 해당 제거 커밋을 revert하는 별도 PR을 만든다.
4. 새 저장소는 문제가 해결될 때까지 읽기 전용 안내를 표시하고 삭제하지 않는다.
5. 링크 PR도 revert 가능한 독립 커밋으로 유지한다.
6. 데이터 손실이 의심되면 새 작업을 중단하고 원본 커밋의 golden-data-lab/ 트리와 새 저장소를 다시 비교한다.
7. 히스토리 추출 작업은 임시 복제본에서만 수행하며 원본 저장소의 main 기록을 강제 갱신하지 않는다.
