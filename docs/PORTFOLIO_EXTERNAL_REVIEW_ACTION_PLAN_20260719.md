# 외부 평가 반영 포트폴리오 보강 계획

작성일: 2026-07-19

## 1. 평가 요약

외부 재평가의 핵심 결론은 다음과 같다.

| 평가 대상 | 현재 판정 |
|---|---|
| 대구 골든타임 대표 프로젝트 | 포트폴리오 공개 가능 |
| GitHub 계정 전체 구성 | 아직 보강 필요 |
| 교육 한 달 차 기준 | 강한 편 |
| 데이터 분석가 지원용 최종 포트폴리오 | 1~2개 축 추가 보강 필요 |

따라서 현재 단계는 새 거대 프로젝트를 다시 만드는 단계가 아니라, 대표 프로젝트를 중심으로 GitHub 계정 첫 화면과 분석가형 데이터랩을 보강하는 단계다.

## 2. 확인된 강점

| 강점 | 포트폴리오에서의 의미 |
|---|---|
| React, FastAPI, 정책분석, EDA, 데이터 계층, 테스트, 문서가 함께 존재 | 단순 웹사이트가 아니라 서비스·데이터·분석 통합 프로젝트로 설명 가능 |
| 시민용 서비스와 정책분석 엔진의 문제 정의가 분리되어 있음 | 사용자와 정책 검토자의 문제를 구분해 설명 가능 |
| 150개 행정동, 25개 기관, 9개 후보, 5,100개 경로 등 검증 수치 존재 | 분석 결과를 정량적으로 제시 가능 |
| VDI, K-Means, 민감도 분석, p-median, MCLP의 사용 이유와 한계 정리 | 방법론을 단순 나열이 아니라 선택 근거로 설명 가능 |
| 테스트와 GitHub Actions 존재 | 결과 파일을 올린 수준이 아니라 재현성과 검증 체계를 제시 가능 |
| AI 활용 범위와 최종 검증 책임을 구분 | AI 보조 개발을 숨기지 않고 전문적으로 설명 가능 |

## 3. 남은 약점

| 약점 | 영향 | 보강 방향 |
|---|---|---|
| GitHub 프로필 README가 없음 | 계정 첫 화면에서 직무 방향과 대표작이 바로 보이지 않음 | `ssg-sak/ssg-sak` 프로필 README 작성 |
| `golden-data-lab`이 아직 계획 단계 | 데이터 분석가형 포트폴리오 축이 미완성으로 보임 | Python EDA, SQL, BI 중 최소 2개 완성 |
| 프로젝트 규모가 큼 | 면접에서 깊은 질문을 받을 가능성이 높음 | 대표 코드, 분석 방법, 검증 체계 Q&A 준비 |

## 4. 즉시 보강할 작업

### 4.1 GitHub 프로필 README

목표는 GitHub 계정 첫 화면에서 다음이 바로 보이게 만드는 것이다.

- 목표 직무: 데이터 분석, 공공데이터 기반 서비스, 백엔드·프론트 구현 경험
- 대표 프로젝트: 대구 골든타임
- 분석 확장 프로젝트: 골든 데이터랩
- 기술 스택: Python, SQL, React, FastAPI, TypeScript, Power BI 예정
- 현재 학습·구축 중인 내용: SQL 분석, EDA, BI 시각화

### 4.2 골든 데이터랩 최소 완성 범위

3주 안에 다음 중 최소 2개를 완성한다.

| 영역 | 최소 완성 기준 |
|---|---|
| Python EDA | 행정동 위험도, 후보지, 접근성 분포를 분석한 노트북과 요약 보고서 |
| SQL 분석 | 주요 테이블 설계, 상위 위험지역, 후보지 비교, 접근성 집계 쿼리 |
| BI 시각화 | Power BI 또는 BI 설계 초안, 지도·위험도·후보지 비교 페이지 |

## 5. 면접 대비 질문

다음 질문은 반드시 본인 언어로 설명할 수 있어야 한다.

- VDI 공식은 왜 그렇게 설계했는가?
- K-Means에서 K=2~5를 검토한 이유는 무엇인가?
- p-median과 MCLP는 어떤 목적이 다른가?
- 5,100개 도로 경로는 무엇을 의미하는가?
- 병상 `null`, `0`, 양수는 서비스에서 어떻게 다르게 표시하는가?
- 왜 HIRA 데이터를 현재 구조에서 제외했는가?
- 24개 기관 누락을 어떻게 발견했고, 25개 기준으로 어떻게 재분석했는가?
- GitHub Actions는 무엇을 검증하는가?
- AI 에이전트는 어디에 활용했고, 최종 판단은 어떻게 했는가?
