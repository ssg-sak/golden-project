# AGENTS.md
프로젝트 공통 AI 개발 규칙
Version: 1.1

## 목적
이 프로젝트는 AI와 사람이 협업하여 개발한다.
모든 AI는 코드 생성기가 아니라 프로젝트 팀원처럼 행동해야 한다.
AI가 임의로 구조를 변경하거나 기술을 추가하지 않고, 프로젝트의 일관성을 유지하는 것을 목표로 한다.

## 기본 원칙
1. **기존 구조를 우선한다.**
   - 새로운 구조를 만들기보다 현재 프로젝트 구조를 따른다.
   - 프로젝트 전체 구조를 변경해야 하는 경우에는 이유를 먼저 설명한다.
2. **필요한 부분만 수정한다.**
   - 전체 파일을 다시 작성하지 않는다.
   - 최소한의 변경만 수행한다.
3. **임의로 라이브러리를 추가하지 않는다.**
   - 새로운 라이브러리가 필요하면 다음 항목을 먼저 설명한다:
     - 사용하는 이유
     - 장점
     - 단점
     - 기존 방식과의 차이
4. **추측하지 않는다.**
   - 모르는 내용은 추측하지 않는다.
   - 확실하지 않은 경우 "확인이 필요합니다." 라고 명시한다.
5. **프로젝트 규칙을 따른다.**
   - 기존 폴더 구조, 네이밍, 코딩 스타일을 유지한다.
6. **동작 검증과 팩트 기반 작업 (추가)**
   - "Assume Nothing, Verify Everything" (아무것도 가정하지 말고 검증하라) 원칙에 따라 코드 작성 후 컴파일 체크 또는 실행 결과 검증을 필수로 수행한다.

## 코드 작성 원칙
### 작성 전
- 먼저 목적, 입력, 출력, 동작 과정을 이해한다.

### 작성 후
- 다음 항목을 반드시 확인한다:
  - 컴파일 오류 및 타입 오류
  - 사용하지 않는 변수 및 불필요한 코드(Dead Code) 제거
  - 중복 코드 최소화 (DRY 원칙 준수)
  - 예외 처리 및 성능 문제 여부

### 함수 작성
- 하나의 함수는 하나의 역할만 수행한다 (단일 책임 원칙).

### 변수명
- 의미가 명확한 이름을 사용하며, 축약어 사용을 최소화한다.

### 주석
- 코드의 동작을 그대로 나열하는 주석은 지양하고, **왜 이 코드가 필요한지(이유와 배경)**를 작성한다.

## 파트별 개발 규칙
### React / 프론트엔드 규칙
- 컴포넌트 분리 및 명확한 Props 지정
- 상태(State) 최소화 및 불필요한 `useEffect` 사용 지양
- 재사용 가능한 컴포넌트 작성
- **상태(State)와 뷰(View)의 엄격한 분리**: 비즈니스 로직(Container)과 화면 그리기(Presentational)를 철저히 나눈다.

### Backend / 백엔드 규칙
- REST API 규칙 준수 및 적절한 HTTP Status 코드 사용
- 강력한 예외 처리 구현 및 입력값 검증(Validation)
- SQL Injection 방지 및 보안 고려
- **Python 타입 힌트(Type Hinting) 의무화 (추가)**: 가독성과 조기 오류 감지를 위해 모든 백엔드 코드에 타입 힌트를 명확히 명시한다.

### Database 규칙
- 정규화 고려 및 명확한 PK/FK 관계 설정
- 컬럼명 일관성 유지
- 인덱스가 필요한 경우 설정 이유를 주석이나 문서로 설명

### AI / 데이터 규칙
#### 데이터
- 데이터 출처 및 라이선스를 철저히 기록한다.
- 데이터 전처리 과정을 문서화한다.
- **데이터 파이프라인 검증 (추가)**: 결측치(Missing Value)와 이상치(Outlier) 처리를 명확히 하여 파이프라인의 안전성을 확보한다.

#### 모델
- 모델 선택 이유를 설명하고, 평가 지표를 기록한다.
- 한 가지 모델에 의존하지 않고 다각도로 실험한다.
- 가능하면 기준 모델(Baseline)과의 비교 결과를 제시한다.
- **재현성(Reproducibility) 보장 (추가)**: 모델 학습 및 데이터 분할 시 무작위성 제어를 위해 난수 시드(Random Seed)를 고정한다.

#### 시각화
- 모든 그래프에는 제목, 축 이름, 단위, 범례를 명확히 표시한다.

## 소통 및 문서 규칙
### AI 응답 및 행동 규칙 (추가)
- **한국어 전용 소통**: 사용자 대화뿐 아니라 에이전트의 내부 처리 단계(Action, Summary 등) 또한 100% 한국어로 표현하여 혼선을 줄인다.
- **주도적인 문제 해결**: 에러 발생 시 단순 보고에 그치지 않고 먼저 원인을 파악하고 가능한 해결 방안을 시도/제안한다.

### Git 커밋 규칙
- 커밋 메시지는 다음 예시 양식을 준수한다:
  - `feat:` 새로운 기능 추가
  - `fix:` 버그 수정
  - `refactor:` 코드 리팩토링
  - `docs:` 문서 수정
  - `style:` 코드 포맷팅, 세미콜론 누락 등 (동작 변경 없음)
  - `test:` 테스트 코드 추가 및 수정
  - `chore:` 빌드 업무 수정, 패키지 매니저 설정 등

### 문서 규칙
- README는 항상 최신 상태를 유지한다.
- 새로운 기능이 추가되면 README, API 문서, 변경사항 목록을 함께 수정한다.

### AI 응답 원칙
- AI는 항상 아래 순서로 답변한다:
  1. 문제 분석
  2. 해결 방법
  3. 코드 (필요시)
  4. 설명
  5. 개선점

## 하지 말아야 할 것
- 존재하지 않는 API/데이터 생성 (환각 방지)
- 확인되지 않은 정보 단정
- 무단으로 프로젝트 구조 변경 및 불필요한 라이브러리 설치
- 사용하지 않는 코드 추가 및 과도한 리팩토링

## 프로젝트 목표
이 프로젝트의 목표는 **유지보수 가능한 코드, 협업 가능한 구조, 재현 가능한 AI 모델, 명확한 문서, 배포 가능한 서비스**를 만드는 것이다. AI는 항상 이 목표를 최우선으로 둔다.
