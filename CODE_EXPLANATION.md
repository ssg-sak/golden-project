# 📘 대구 골든타임 코드 학습·설명서 (Code Explanation)

본 문서는 프로젝트의 코드 베이스와 아키텍처 철학을 이해하기 위한 설명서입니다. 
AI 에이전트와 개발 팀원이 일관된 구조로 개발할 수 있도록 기준을 제시합니다.

## 1. 프론트엔드 아키텍처: Feature-Sliced Design (FSD) 기반

이 프로젝트의 React 프론트엔드는 일반적인 단일 `components/` 폴더 대신, **FSD (Feature-Sliced Design)** 개념을 차용하여 레이어별로 철저하게 분리되어 있습니다.

### 📁 디렉토리 구조 (`frontend/src/`)
- **`app/`**: 애플리케이션 진입점, 전역 설정, Providers, Router 등 가장 바깥쪽 레이어
- **`widgets/`**: 여러 UI 컴포넌트가 결합된 독립적인 비즈니스 뷰 블록 (예: 사이드바, 지도 캔버스, 우측 상세 패널)
- **`shared/`**: 프로젝트 전반에서 재사용 가능한 순수 UI 컴포넌트(Presentational), 유틸리티 함수, 커스텀 훅(Hooks)
- **`types/`**: 애플리케이션 전반에서 사용되는 공통 타입스크립트 인터페이스 및 타입 정의
- **`data/`**: API 통신, 상태 관리 및 데이터 패칭 레이어 (Container/Controller 역할)
- **`assets/`**: 이미지, 아이콘, 폰트 등 정적 리소스 파일

### 🏗️ 뷰(View)와 상태(State)의 엄격한 분리
- **바이브코딩 및 단일 책임 원칙**: 컴포넌트 파일 하나가 너무 많은 책임을 지면 유지보수가 어려워집니다. 
- 데이터 로깅, API 통신 등의 상태 변환 로직은 `data/`나 `widgets/` 상위 컨테이너가 담당합니다.
- `shared/` 내부의 컴포넌트는 오직 `props`만 받아 화면에 그리는 프레젠테이셔널 역할만 수행합니다.

## 2. 백엔드 아키텍처 (FastAPI)

- 데이터 무결성을 위해 카카오맵 API 및 공공 데이터(HIRA) 연동부를 서비스 레이어로 철저히 캡슐화합니다.
- 외부 API 장애 상황을 대비해 폴백(Fallback) 캐시 메커니즘을 내장하고 있습니다.

## 3. 개발 룰 및 에이전트 원칙

프로젝트 루트의 `.agents/AGENTS.md`에 정의된 룰을 최우선으로 따릅니다:
1. DRY (Don't Repeat Yourself) 원칙 준수
2. 의존성 추가나 코드 작성 시 기존의 `shared/` 영역 우선 탐색
3. 항상 검증하고, 스스로 체크하는 사후 검증(Terminal Verification) 의무화
